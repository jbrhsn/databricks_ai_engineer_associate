**Q280: A team notices that their production RAG application has `groundedness = 0.92` but `correctness = 0.45`. They also measure `chunk_relevance = 0.34` and `context_sufficiency = 0.38`. Which layer is failing, and what is the most likely next investigation step?**

   A. The LLM generation layer is hallucinating; reduce temperature to 0.1.
   B. The retrieval layer is the bottleneck — chunk_relevance and context_sufficiency are both low, meaning the retrieved chunks are not addressing the question, so the answer cannot be correct even though it is grounded in whatever was retrieved.
   C. The embedding model is corrupt; rebuild all vector indices.
   D. The groundedness metric is misconfigured; groundedness and correctness cannot diverge this widely.

A: **B**. High groundedness (0.92) means the LLM is faithfully reproducing what it found in the retrieved context — so if the answer is wrong, the context itself must be insufficient or irrelevant. Chunk_relevance = 0.34 confirms that the retrieved chunks are not addressing the question; context_sufficiency = 0.38 confirms the retriever is not fetching information sufficient to answer correctly. The generation layer is working as designed; the retrieval layer needs debugging — check embedding model quality, chunk size, index freshness, or query representation. Option A is wrong because high groundedness rules out hallucination as the cause. Option C is unsupported. Option D is wrong — high groundedness with low correctness is a documented failure mode indicating retrieval failure, not a misconfiguration.

**Q283: An ML team wants to measure whether their agent is grounded in retrieved context AND produces factually correct answers for production monitoring. They have no labeled ground-truth dataset for production queries. Which TWO judges should they use for continuous production monitoring?**

   A. `groundedness` (requires no ground truth; detects hallucination relative to retrieved context)
   B. `correctness` (requires labeled ground truth; tests factual accuracy)
   C. `relevance_to_query` (requires no ground truth; confirms the response addresses the question)
   D. `context_sufficiency` (requires labeled ground truth; tests retrieval completeness)
   E. `chunk_relevance` (requires no ground truth; scores each retrieved chunk's relevance)

A: **A and C**. **A qualifies** because groundedness requires only the request, response, and retrieved_context columns — no ground-truth labels needed. It detects hallucination by checking whether every claim in the response is traceable to the context, making it ideal for production monitoring where labeled answers don't exist. **C qualifies** because relevance_to_query requires only the request and response — no ground truth — and confirms the response actually addresses the user's question rather than being off-topic or evasive, which is a critical production-safety check. **B is wrong** — correctness requires `expected_facts` or `expected_response`, which don't exist for production queries without a labeling infrastructure. **D is wrong** — context_sufficiency requires ground truth. **E is wrong** — while chunk_relevance doesn't require ground truth, it only scores retrieval quality, not generation quality; the scenario needs both retrieval and generation assessment, which is why A and C together provide full coverage.

**Q287: Which THREE of the following are reasons to prefer using SME-provided `expectation` assessments from the Review App over `feedback` assessments when building an evaluation dataset?**

**Q288: An ML engineer sets `sampling_fraction=0.1` on an inference table (logs 10% of endpoint requests) and registers an MLflow production monitor with `sample_rate=0.2` on the Correctness judge. What fraction of the actual endpoint traffic will be evaluated by the Correctness judge?**

   A. 10%
   B. 20%
   C. 2%
   D. 30% (rates are additive)

A: **C**. The two sampling parameters are multiplicative, not additive. The inference table logs 10% of traffic (9/10 of requests don't log), and then the monitor samples 20% of those logged traces. Effective coverage = 0.1 × 0.2 = 0.02 = 2% of actual traffic. This is a critical detail — engineers often set both parameters independently without realizing they compound, resulting in far-too-sparse evaluation. A common mistake is thinking `sample_rate` applies to all traffic — it applies only to traces that were already logged by the inference table.

   A. A checkpointer is mandatory for all LangGraph agents to enable production deployment.
   B. A checkpointer is optional at compile time; if omitted, the graph can still run but multi-turn memory and resumable execution are not available.
   C. The checkpointer persists graph state to a backend (e.g., SQLite, PostgreSQL, or in-memory) so that if a node fails, the graph can resume from that checkpoint rather than restarting from the beginning.
   D. Checkpointers are configured at graph compile time via `graph.compile(checkpointer=...)` and enable the graph to save intermediate state after each node completes.

A: **B and D**. **B qualifies** because checkpointers are optional at graph compile time — the graph runs fine without one, but it loses resumable-execution capability; every failure starts from scratch. **D qualifies** because the checkpointer is a parameter to `compile()` and is the mechanism that persists state snapshots, enabling recovery from mid-graph failures. **A is wrong** — checkpointers are optional, not mandatory; many simple agent deployments omit them because the overhead is not worth it for single-turn use cases. **C is wrong** — while checkpointers do persist state, they do not enable automatic resume-on-failure without explicit code; you must manually re-invoke with the thread ID that recovers the checkpoint. **E is wrong** — LangGraph state is mutable; it is passed by reference and updated in-place by node functions via `state["key"] = value`, which is exactly what checkpointers capture.
