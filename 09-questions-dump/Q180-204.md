**Q180: A Generative AI Engineer is designing a RAG vs. fine-tuning decision matrix for a customer support application. The knowledge base contains frequently updated company policies (weekly changes), and the team has a small labeled dataset (50 Q&A pairs) of support conversations. Which combination of factors MOST strongly favors RAG over fine-tuning for this application?**

   A. The small labeled dataset is insufficient for fine-tuning and frequent policy updates cannot be deployed via model retraining.
   B. The team has only 50 labeled Q&A pairs, which guarantees that RAG will produce higher quality than fine-tuning.
   C. Fine-tuning always produces deterministic outputs while RAG introduces variability.
   D. RAG allows the system to cite specific policy documents as the source of answers.

A: **A and D qualifies.** **A qualifies** because RAG is the natural fit when (1) knowledge is updatable at runtime without retraining and (2) labeled training data is sparse (50 pairs is too small for supervised fine-tuning, which typically requires thousands to hundreds of thousands of examples). Fine-tuning on such a small dataset would lead to overfitting and requires a full retraining cycle for every policy change. **D qualifies** because RAG's architectural foundation — retrieving and surfacing source documents — enables the application to return citations showing which policy document supports the answer, satisfying audit and transparency requirements that are often mandated in customer support contexts. B is wrong because a small labeled dataset alone does not guarantee RAG superiority — it depends on whether RAG has access to a high-quality knowledge base; without one, both approaches fail. C is wrong because temperature and sampling parameters control determinism for both RAG and fine-tuned models equally; RAG is not inherently more variable.

**Q181: Which TWO of the following are valid strategies for managing context window constraints in a long-document RAG application where the average retrieved context is 12,000 tokens but the LLM's context window is 8,000 tokens?**

   A. Increase the LLM's maximum output tokens parameter to expand the available context window.
   B. Use the Parent Document Retriever pattern: index small chunks (e.g., 256 tokens) for precise retrieval, but return the larger parent chunks (e.g., 1024 tokens) to the LLM, then implement additional filtering by metadata or re-ranking to stay within the context budget.
   C. Implement a re-ranking step to select the most relevant subset of retrieved chunks before passing them to the LLM, keeping the highest-precision results within the context budget.
   D. Switch to a smaller embedding model that produces lower-dimensional vectors, which reduces context size.
   E. Implement hybrid search with BM25 to reduce the number of retrieved chunks.

A: **B and C**. **B qualifies** because the Parent Document Retriever pattern solves exactly this problem by decoupling the granularity of retrieval (small, precise chunks) from the granularity of context passed to the LLM (larger, richer chunks with surrounding context). If individual retrieved chunks are too large, the pattern returns only the specified number of parent chunks after retrieval, keeping the final context within the budget. **C qualifies** because re-ranking filters the broad initial retrieval set down to the most relevant subset before context assembly, directly reducing total tokens passed to the LLM while preserving the highest-precision results. **A is wrong** because `max_tokens` controls only the *output* limit, not the input context window; it cannot expand how much text the LLM can read. **D is wrong** — embedding dimensionality does not affect context window size; it affects vector size in memory, not the number of tokens in the prompt. **E is wrong** because reducing the number of retrieved chunks (via hybrid search or any other method) trades recall for staying under budget, but it is not a direct context-window strategy like re-ranking or the Parent Document Retriever.

**Q182: A Generative AI Engineer is building a multi-step reasoning agent. The application requires the agent to decompose complex customer queries into sub-tasks, call tools sequentially, and produce a final synthesis. Which of the following design patterns BEST aligns with these requirements, and why?**

   A. A single LangChain chain that calls tools via `RunnableLambda` without explicit state management.
   B. A LangGraph StateGraph where each node represents a step (decompose, retrieve, reason, synthesize), and conditional edges route between steps based on the current state.
   C. A Databricks SQL loop that iteratively calls `ai_query()` for each reasoning step.
   D. A REST API that calls a different microservice for each reasoning step, with no state coordination.

A: **B.** LangGraph's StateGraph is purpose-built for multi-step workflows with state coordination. It allows the engineer to define distinct nodes for decomposition, retrieval, reasoning, and synthesis, with conditional edges routing based on the state at each step (e.g., "if the query was successfully decomposed, proceed to retrieval; else loop back to decompose"). The shared state dict persists information from one step to the next, and checkpointing preserves the entire execution history for multi-turn conversations. This structure is auditable (each step is visible in traces), testable (nodes are independent functions), and extensible (adding a new reasoning step is a new node + edge). **A is wrong** because a simple chain lacks state coordination; if a step fails or needs to conditionally branch, the chain cannot express that without significant workarounds. **C is wrong** because `ai_query()` is designed for batch SQL inference, not stateful multi-step agentic reasoning. **D is wrong** because a microservice per step creates deployment and debugging complexity and loses the observability and memory sharing that LangGraph provides.

**Q183: You are designing the prompt for a RAG application that must avoid hallucinating outside the retrieved context. Which TWO of the following prompt engineering techniques are MOST effective at enforcing grounding?**

   A. Include explicit instructions in the system prompt: "If the context does not contain the answer, say 'I don't know' rather than guessing."
   B. Set the model's temperature to 0.0 to eliminate all randomness in token generation.
   C. Provide a few-shot example demonstrating refusal when the context is insufficient.
   D. Use a larger embedding model, which will retrieve more complete documents.
   E. Add a guardrail that scans the output for factual consistency with the retrieved context.

A: **A and C**. **A qualifies** because explicit instructions in the system prompt directly inform the LLM's behaviour during generation. A grounding instruction like "answer only from the provided context; if the context is insufficient, decline to answer" is the primary mechanism to reduce hallucination from parametric memory, and it is recognized in production systems as an essential first layer. **C qualifies** because few-shot examples demonstrate the exact behaviour the LLM should replicate — showing a correct refusal when context is absent is more concrete and actionable than a generic instruction alone. The LLM pattern-matches on the examples and is more likely to follow the refusal pattern when faced with an out-of-context query. **B is wrong** because temperature controls randomness in sampling, not grounding; even at temperature 0.0, an LLM can confidently generate plausible-sounding but unsupported statements if the system prompt does not forbid it. **D is wrong** — embedding model size does not affect prompt grounding; it affects retrieval quality, which is an upstream concern. **E is wrong** — guardrails are a valid secondary safety layer but are not a *prompt engineering* technique; the primary technique is the system prompt instruction itself.

**Q184: A production RAG application using a 4K-token context window processes documents averaging 8,000 tokens. The team is observing frequent failures due to context truncation. What is the MOST direct root-cause diagnosis and corrective action?**

   A. The embedding model is too small; switch to a larger embedding model to encode more semantic information.
   B. The retrieved context (8K tokens average) exceeds the LLM's available context budget after accounting for the system prompt, user query, and output tokens; implement a semantic chunking or re-ranking strategy to reduce retrieved context size to fit within the budget.
   C. The system prompt is too verbose; compress the system prompt to save context tokens.
   D. Increase the model's `max_tokens` parameter to allow longer outputs, which will provide more space for context.

A: **B.** The diagnosis is straightforward: 8K retrieved context + system prompt (~500 tokens) + user query (~200 tokens) + output budget (~500 tokens) = ~9,200 tokens total, which exceeds the 4K context window. **B correctly identifies the root cause (context size mismatch)** and proposes the two primary mitigations: semantic chunking to retrieve fewer, more precise chunks, or re-ranking to filter the retrieved set to only the most relevant results. Both keep the application within budget. **A is wrong** — embedding model size affects the quality of retrieval ranking but does not change the number of tokens in the retrieved text or the model's context window. **C is wrong** — while compressing the system prompt could save tokens, the real problem is a 2× mismatch between available window and document size; prompt compression alone is insufficient. **D is wrong** — `max_tokens` controls only the *output* limit, not the input context window; it cannot solve an input size problem.

**Q186: A Generative AI Engineer is chunking a technical reference manual with clear section headers (## API Reference, ### Authentication, #### Token Endpoint). The team wants chunks to be individually retrievable as complete topics but also to preserve the hierarchical context (e.g., knowing that a chunk belongs under "API Reference > Authentication"). Which chunking strategy BEST satisfies both constraints?**

   A. Use RecursiveCharacterTextSplitter with a `chunk_size` of 500 tokens and `chunk_overlap` of 50 tokens.
   B. Use MarkdownHeaderTextSplitter to split by header hierarchy, injecting header metadata into each chunk's metadata dict, followed by RecursiveCharacterTextSplitter on oversized sections.
   C. Concatenate all sections into a single large chunk and rely on the embedding model to understand the hierarchy.
   D. Use a fixed-size CharacterTextSplitter with zero overlap to minimize storage cost.

A: **B.** MarkdownHeaderTextSplitter solves this exactly: it scans for Markdown headers (`##`, `###`, etc.), creates one chunk per section, and crucially injects the full header path (e.g., `{"Header 2": "API Reference", "Header 3": "Authentication", "Header 4": "Token Endpoint"}`) into the chunk's metadata without polluting the chunk text itself. This metadata is queryable in Databricks Vector Search, enabling filtering by section. When a section exceeds the size limit, a recursive pass splits it while preserving the metadata from the parent headers. **A is wrong** because RecursiveCharacterTextSplitter treats the text as unstructured; it has no concept of headers and will split across section boundaries, losing the structural information. **C is wrong** because embedding models do not parse document structure; a single large chunk loses the ability to filter by section and degrades retrieval precision. **D is wrong** — CharacterTextSplitter with zero overlap discards context at boundaries, causing sentences to be split mid-thought, and fixed-size splitting ignores semantic boundaries.

**Q187: Which TWO of the following Delta table schema design patterns are MOST important when storing chunked documents for RAG retrieval?**

   A. Store one row per document (entire document as a single large string column) to minimize row count and storage.
   B. Store one row per chunk, with columns for chunk text, metadata, and a unique chunk ID for tracking, enabling chunk-level retrieval and filtering.
   C. Include a `source_document_id` column so chunks can be traced back to their source and filtered by document-level attributes.
   D. Store chunks in a nested struct column with all metadata embedded, which reduces the number of columns and simplifies schema management.
   E. Store embeddings directly in the Delta table as dense vector columns so the vector index can be synced automatically.

A: **B and C**. **B qualifies** because storing one row per chunk (not per document) is the foundational schema requirement for RAG. Each chunk is independently queryable, retrievable, and filterable. Metadata columns and unique IDs enable citation, provenance tracking, and filtering in Databricks Vector Search. One-row-per-document designs prevent chunk-level retrieval entirely. **C qualifies** because `source_document_id` (or similar lineage columns) enables document-level filtering at retrieval time (e.g., "return only chunks from Q3 2026 earnings reports") and enables downstream citation (the application can tell users which original document the chunk came from). This is essential for audit and compliance. **A is wrong** — one row per document is incompatible with chunk-level retrieval and destroys filtering granularity. **D is wrong** — while nested structs are valid, they offer no advantage over flat columns for metadata; they add complexity without solving a problem. **E is wrong** — embeddings are typically generated by the Vector Search index's embedding endpoint (in Delta Sync mode) or pre-computed and upserted via the Direct Vector Access API; storing them redundantly in the source Delta table wastes space and complicates sync logic.

**Q188: A chunking pipeline currently uses token-based splitting (TokenTextSplitter with token count measured by `tiktoken.encoding_for_model("text-embedding-3-small")`), but the team is migrating to a different embedding model with a smaller context window (512 tokens vs. 8192 tokens). What change to the chunking configuration is REQUIRED to prevent truncation errors at retrieval time?**

   A. Increase `chunk_size` to 2048 tokens to compensate for the smaller embedding model.
   B. Change the `length_function` parameter in TokenTextSplitter to use the tokenizer corresponding to the new embedding model.
   C. Switch from TokenTextSplitter to RecursiveCharacterTextSplitter to avoid token-level precision.
   D. No change is needed; the embedding model's context window does not affect chunking configuration.

A: **B.** The `length_function` parameter controls which tokenizer measures chunk size. When the embedding model changes, the tokenizer changes — `text-embedding-3-small` uses OpenAI's tokenizer, while the new embedding model uses its own BPE encoder. Different tokenizers produce different token counts for the same text; the same 500-character passage might be 80 tokens in one and 120 in another. To ensure chunks stay within the new embedding model's 512-token limit, the `length_function` must be updated to use the new model's tokenizer. Failing to do so can result in chunks that exceed the embedding model's limit, causing truncation or API errors. **A is wrong** — increasing chunk size makes the problem worse, not better; chunks must *decrease* in size to fit the smaller window. **C is wrong** — switching to RecursiveCharacterTextSplitter based on character count, not tokens, is less precise and doesn't solve the token-limit problem. **D is wrong** — the embedding model's context window is exactly what determines the maximum chunk size, since chunks that exceed the limit will fail at embedding time.

**Q192: Which TWO of the following statements accurately describe how reducers work in a LangGraph StateGraph state definition?**

   A. A reducer is a function called by LangGraph whenever a node returns a partial update; it determines how the new value is merged with the existing value for that channel.
   B. Reducers eliminate the need for node functions to return partial dicts; nodes must return the full state object instead.
   C. The `add_messages` reducer appends new messages to an existing list rather than replacing it, solving the multi-turn message accumulation problem.
   D. Without a reducer, a plain `list` or `dict` field is overwritten entirely by any update, losing prior values unless explicitly copied.
   E. Reducers are optional; if not specified, all fields are treated as overwrite-only.

A: **A and C**. **A qualifies** because that is the exact definition of a reducer in LangGraph's channel system. When a node returns `{"messages": [new_msg]}`, the reducer for the `messages` channel is invoked with `(existing_messages, new_messages)` and returns the merged result. Without a reducer, the update would simply overwrite. **C qualifies** because `add_messages` is the canonical example: when applied to a `messages` field, `add_messages(old_list, new_messages)` appends the new messages, solving the critical problem of accumulating conversation history across multiple node invocations. Nodes still return partial dicts, and reducers control how partials are merged. **D is true as a statement but is not the most direct description of how reducers work; the question asks about reducer mechanics, not consequences.** **E is wrong** — reducers are not optional; every field has a default reducer (usually overwrite), but specifying a custom reducer like `add_messages` is essential for multi-turn state.

**Q198: Which TWO of the following correctly describe the role of MLflow Tracing in debugging a LangGraph agent?**

   B. MLflow Tracing can only capture LLM call spans; tool calls and intermediate node outputs are invisible to tracing.
   C. With `mlflow.langchain.autolog()` enabled, LangGraph agent invocations are automatically instrumented with spans without requiring manual code changes to individual nodes.
