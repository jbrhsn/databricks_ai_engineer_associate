**Q255: A Generative AI Engineer is auditing a customer-support chatbot that answers questions using internally-governed data. The team wants to ensure that confidential customer names, email addresses, and support ticket IDs never reach the model or appear in inference logs, even if a user includes them in their query. Which two-layer approach BEST addresses this requirement?**

    A. Configure AI Gateway PII filtering to mask emails and SSNs only, then rely on the model's instruction following not to repeat PII in responses
    B. Apply application-layer input sanitization using regex patterns before logging or prompt assembly, THEN use AI Gateway PII filtering as a centralized backstop at the endpoint
    C. Store PII only in Unity Catalog tables with column masks, and skip any filtering at the application or endpoint layers
    D. Use tokenization to replace identifiers in the source documents before chunking and indexing, then add output masking at the serving endpoint

A: **B and D**. **B qualifies** because application-layer sanitization (regex-based masking of emails, phone numbers, and custom identifiers) removes sensitive data before it reaches logs, traces, and prompt builders, while AI Gateway serves as a second enforcement point for any PII missed by the first layer. This defence-in-depth approach prevents PII from entering both upstream processes and downstream model I/O. **D qualifies** because tokenizing identifiers before chunks are created prevents raw PII from entering the vector index — a critical propagation point — and endpoint-level masking adds a final safety net for any PII that escapes earlier controls. **A is wrong** because relying on model instruction-following for privacy is not a control; models can and often do repeat sensitive information. **C is wrong** because it addresses only the source data layer and ignores runtime exposure in prompts and logs. **E is wrong** because disabling logging hides evidence but does not prevent leakage; it also eliminates audit trails needed for compliance.

    D. Use `mlflow.pyfunc.PythonModel` with a `predict(context, model_input)` method that reconstructs the agent from the logged graph definition and input; pass conversation history as part of the input, not stored state

A: **D**. The correct pattern is to log the agent's definition (graph, tools, system prompt) as artifacts and reconstruct it fresh in each `predict()` call. Conversation history is passed in `model_input` (e.g., as part of a messages array), not stored in model state. This ensures the serving replica is stateless and thread-safe — essential for horizontal scaling. Option A (fresh state every time) is correct in principle but loses conversation context unless history is passed in. Option B (log memory as artifact) conflates model definition with runtime state. Option C (Secrets) is inefficient. Option E skips MLflow governance. The PyFunc wrapper should encapsulate agent logic, not runtime memory.

**Q279: Which type annotation pattern on a tool's return value BEST helps a LangGraph agent predict the structure of tool outputs and decide how to incorporate results into subsequent reasoning?**

    A. Using untyped `dict` return types forces the LLM to infer output structure from examples
    B. Using `Pydantic` models or structured output annotations (e.g., `list[str]`, `dict[str, float]`) on the tool's return type signals the LLM the expected structure via the tool schema
    C. Using `Any` type allows maximum flexibility for the tool's output
    D. Type annotations on tools are ignored by LangGraph; structure is inferred only from the tool name

A: **B**. Type annotations on tool return values are translated into the tool's JSON schema (visible in the LLM's function calling interface). When a tool is annotated with `list[str]` or a `Pydantic` model, the LLM receives structured schema information about what to expect, improving its reasoning about how to use the tool's output in the next step. Structured return types (especially Pydantic models) also validate and normalize tool outputs before they re-enter the agent loop. Option A (untyped) offers no schema guidance. Option C (`Any`) lacks precision. Option D is wrong — annotations absolutely matter. Option E (all `str`) discards semantic richness. Well-typed tools produce more reliable agents because the LLM and type validators are aligned on data structure contracts.
