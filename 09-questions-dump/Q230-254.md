**Q230: A Generative AI Engineer is deciding between Delta Sync and Direct Vector Access index types for an AI Search implementation. The source data is a rapidly-changing stream of real-time transaction data arriving every second via a Kafka connector, and the data is stored in a Unity Catalog Delta table with Change Data Feed enabled. Which index type is most appropriate, and why?**

   A. Delta Sync with `pipeline_type="TRIGGERED"` — TRIGGERED mode automatically reacts to CDF events and achieves sub-second freshness with minimal cost.

   D. Neither; real-time data must use a separate vector database outside of Databricks.

A: **C**. For rapidly-changing data where sub-second or few-second latency is required, Delta Sync with `pipeline_type="CONTINUOUS"` is the correct choice. CONTINUOUS mode spins up a Spark Structured Streaming pipeline that continuously polls the Delta table's Change Data Feed and propagates new/updated rows into the HNSW index within seconds. While this mode is more expensive than TRIGGERED (because it maintains a persistent cluster), it is the only way to achieve near-real-time freshness for high-frequency updates within the Delta Sync framework. Option A is wrong because TRIGGERED mode requires manual `index.sync()` calls—it does not automatically react to table changes, making it unsuitable for real-time data streams. Option B is wrong because Delta Sync with CONTINUOUS is the production-grade solution; Direct Vector Access would require the engineer to build and maintain a separate embedding + upsert pipeline, duplicating work. Option D is wrong because Databricks AI Search fully supports streaming data via CONTINUOUS Delta Sync.

**Q233: A Model Serving endpoint serving a RAG agent has been created with `scale_to_zero_enabled=True`. After 8 hours of no incoming requests, a user submits a query. What is the most likely behavior, and what is the correct mitigation for a production use case with a sub-100ms SLA?**

A: **B**. When `scale_to_zero_enabled=True`, Databricks suspends all compute replicas after an idle period to save cost. Option A is wrong because scale-to-zero does not permanently disable the endpoint; it will resume serving once it spins back up. Option C is wrong because `scale_to_zero_enabled=True` explicitly disables warm-standby behavior. Option D is wrong because the AI Playground is a development/testing surface, not a failover mechanism for production endpoints.

Option C is wrong because parent-document retrieval usually *increases* token usage (larger context), though the number of vectors indexed is the same.
