**Q1: A Generative AI Engineer has created a RAG application to look up answers to questions about a series of fantasy novels. They want to methodically choose the best chunking strategy and parameters. Which TWO strategies should they take?**

O: a) Change embedding models and compare performance.
   b) Add a classifier for user queries that predicts which book will best contain the answer. Use this to filter retrieval.
   c) Choose an appropriate evaluation metric (such as recall or NDCG) and experiment with changes in the chunking strategy, such as splitting chunks by paragraphs or chapters. Choose the strategy that gives the best performance metric.
   d) Pass known questions and best answers to an LLM and instruct the LLM to provide the best token count. Use a summary statistic (mean, median, etc.) of the best token counts to choose chunk size.
   e) Create an LLM-as-a-judge metric to evaluate how well previous questions are answered by the most appropriate chunk. Optimize the chunking parameters based upon the values of the metric.

A: **C and E**. Systematic evaluation using metrics like recall and NDCG combined with LLM-as-a-judge evaluation provides both quantitative and qualitative assessment of chunking strategy effectiveness. This combination is essential because recall/NDCG measure retrieval accuracy (which chunks were found), while LLM-as-a-judge measures whether the retrieved chunks actually produce good final answers. **A is wrong because** changing embedding models introduces a confounding variable—if performance changes, you cannot distinguish between chunking improvements and embedding model differences. **B is wrong because** a classifier that predicts which book will answer a question is a workaround that requires manual domain expertise rather than letting the system learn optimal chunking through empirical measurement. **D is wrong because** using LLM-generated token count summaries has no connection to actual retrieval or generation quality—LLM guesses about token counts do not reflect how well different chunk sizes serve the downstream pipeline.

   c) Faithfulness/groundedness of generated answers — whether responses are supported by the retrieved context rather than hallucinated

Leaderboard scores (B) are benchmark results, not production metrics. Throughput under load (D) is an infrastructure concern, not a measure of answer correctness.

**Q4: A Generative AI Engineer is building a system that suggests the best matched employee for newly scoped projects based on availability and profile match. How should they architect the system?**

O: a) Create a tool for finding available team members. Embed all project scopes into a vector store, perform retrieval using team member profiles.
   b) Create a tool for finding availability and another for extracting keywords. Iterate through available members' profiles performing keyword matching.
   c) Create a tool for finding available team members and another that calculates similarity score. Iterate through members and rank by best score.
   d) Create a tool for finding available team members. Embed team profiles into a vector store and use project scope with filtering to perform retrieval.

A: **D**. Embedding team member profiles into a vector store and using project scope as a query with availability filtering is the ideal architecture because it handles both unstructured text matching (via vector search on profiles) and structured constraints (availability) simultaneously. This approach scales well as teams and projects grow, and vector search naturally captures semantic similarity between project requirements and team skills. **A is wrong because** it inverts the embedding direction—embedding project scopes (which change frequently) instead of profiles (which are relatively stable) creates unnecessary index rebuilds; worse, retrieving against team profiles is the wrong semantic match. **B is wrong because** keyword matching on team profiles is brittle and rigid; it misses semantic skills that aren't captured by exact pre-defined keywords (e.g., "data pipeline experience" wouldn't match queries for "ETL")—real profiles have nuanced skill descriptions that require semantic understanding. **C is wrong because** manually iterating through members with calculated scores is functionally equivalent to vector search but less efficient at scale and doesn't leverage the vector database infrastructure—it's essentially reinventing vector search in Python, which sacrifices both performance and maintainability.

**Q7: A Generative AI Engineer is building a RAG application for SnoPen AI that answers questions about internal company documents. Source documents contain a mix of relevant company content and irrelevant content (competitor news, entertainment articles). Which approach MOST effectively prevents irrelevant content from surfacing in generated answers?**

O: a) Include in the system prompt that the LLM should only answer questions about SnoPen AI
   b) Filter and remove irrelevant documents during the ingestion pipeline before they are chunked and indexed into the vector store
   c) Increase the number of retrieved chunks so the relevant content is more likely to appear
   d) Consolidate all documents into a single chunk to reduce noise

A: **B**. Pre-ingestion filtering is the most effective approach: if irrelevant documents are never indexed, they can never be retrieved and injected into the LLM's prompt window. A system-prompt instruction (A) is a weak backstop — it cannot prevent the retriever from fetching irrelevant chunks and consuming context window space. Retrieving more chunks (C) amplifies the problem. Single-chunk consolidation (D) destroys retrieval granularity.

**Q8: A Generative AI Engineer has ingested unstructured documents and chunked them by section. They want to store chunks in a Vector Search index. Current dataframe has: (i) original document file name (ii) array of text chunks. What is the most performant way?**

O: a) Split into train and test set, create a unique identifier for each document, then save to a Delta table.
   b) Flatten the dataframe to one chunk per row, create a unique identifier for each row, and save to a Delta table.
   c) Create a unique identifier for each document, then save to a Delta table.
   d) Store each chunk as an independent JSON file in Unity Catalog Volume with document section name as key.

A: **B**. Flattening the dataframe to one chunk per row with unique identifiers creates the optimal schema for vector indexing because the Vector Search engine expects tabular data where each row is independent and searchable—one chunk per row allows the engine to assign a unique ID and embedding to each chunk, enabling precise retrieval and ranking. Keeping chunks in an array structure (one document per row) would force the vector database to handle nested data or require custom exploding logic before indexing, adding complexity and reducing query precision. **A is wrong because** splitting into train/test sets before ingestion introduces unnecessary data fragmentation and doesn't address the core challenge of preparing the chunks for indexing—train/test splitting happens after data is stored, not before ingestion. **C is wrong because** creating a unique identifier at the document level (not chunk level) loses the ability to retrieve specific chunks; the Vector Search index needs to return relevant individual chunks, not entire documents, which risks overwhelming the LLM's context window with irrelevant content. **D is wrong because** storing each chunk as an independent JSON file in Unity Catalog Volumes is unstructured and unindexed—Vector Search cannot efficiently search or rank JSON files without first materializing them into a queryable index backed by Delta, making this approach operationally inefficient and incompatible with Vector Search's indexing requirements.

**Q11: A Generative AI Engineer developed an LLM application using provisioned throughput Foundation Model API. Volume is insufficient for provisioned throughput. Which strategy ensures cost-effectiveness?**

O: a) Switch to using External Models instead.
   b) Switch to pay-per-token Foundation Model API serving, which has no fixed upfront capacity commitment and scales cost with actual usage.
   c) Change to a model with fewer parameters to reduce hardware constraints.
   d) Throttle incoming batch requests manually to avoid rate limiting.

A: **B**. Pay-per-token billing has no fixed upfront capacity cost — you pay only for tokens consumed. For a workload whose volume is insufficient to justify provisioned throughput (which requires a committed DBU allocation), pay-per-token avoids stranded capacity costs. Note: pay-per-token does not provide a cost *ceiling* or *guarantee* — costs scale with usage. External Models (A) would introduce external API dependencies and data-residency concerns. Throttling manually (D) is error-prone and does not reduce per-token cost.

**Q14: A Generative AI Engineer is designing a chatbot for a gaming company. Which metric would help increase user engagement and retention?**

O: a) Response consistency with a fixed template structure
   b) Diversity of responses
   c) High confidence in responses, even when uncertain
   d) Identical responses repeated for all similar questions

A: **B**. Diverse responses prevent user boredom and increase engagement by varying conversation style and content, which directly improves retention as users find the interaction fresh and valuable rather than repetitive. Different response wordings for similar questions, varied tone, and contextually appropriate examples keep the experience dynamic and mentally stimulating for long-term users. **A is wrong because** response consistency with a fixed template structure (response consistency/template rigidity) makes interactions feel robotic and mechanical—users quickly recognize the pattern and lose interest in a predictable, machine-like bot. **C is wrong because** high confidence in responses even when uncertain is dishonest and damages trust when the model is proven wrong later—users abandon tools that confidently lie to them. **D is wrong because** identical responses repeated for all similar questions (exact repetition) breed monotony, which is antithetical to engagement and retention—users quickly realize the bot has limited value if it cannot provide varied perspectives on topics.

**Q17: A Generative AI Engineer is building a RAG application from PDFs containing both text and embedded images. They want to extract text content using the least amount of custom parsing code. Which Python package is BEST suited?**

O: a) PyMuPDF (fitz) — low-level PDF rendering library requiring custom layout parsing logic
   b) pdfplumber — table-focused PDF extraction requiring per-document configuration
   c) unstructured — unified document parsing library that handles mixed-content PDFs with minimal code
   d) camelot — specialist PDF table extraction with limited support for general text and images

A: **C**. The `unstructured` library provides a high-level unified interface that handles text and image-containing PDFs with minimal custom code — it is the lowest-effort path for mixed-content PDF ingestion in RAG pipelines. `PyMuPDF` (A) and `pdfplumber` (B) are lower-level and require more custom layout logic. `camelot` (D) is optimised for tables only. On Databricks, `unstructured` integrates directly with LangChain document loaders used in ingestion pipelines.
