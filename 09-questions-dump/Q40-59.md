**Q40: Which TWO of the following are correctly matched — an Agent Bricks capability to the business scenario it is designed to address?**

   A. Knowledge Assistant → A support team wants a low-code Q&A assistant over internal policy documents without writing a custom RAG pipeline
   B. Multi-Agent Supervisor → A team needs to coordinate a SQL analytics agent, a document-search agent, and a ticket-creation agent under a single conversational interface
   C. Information Extraction → A data team wants low-latency semantic search over a 500-million-vector embedding corpus
   D. Knowledge Assistant → A team needs to pull structured fields (invoice number, date, total) from scanned vendor invoices at scale
   E. Multi-Agent Supervisor → A data engineer wants to apply an LLM to classify sentiment on rows in a Delta table using a SQL query

A: **A and B**. A: Knowledge Assistant is designed exactly for low-code document Q&A without custom RAG pipeline development. B: Multi-Agent Supervisor orchestrates and routes requests across multiple specialised sub-agents under one interface. C is wrong — semantic search over embeddings is Vector Search / AI Search, not Information Extraction. D is wrong — structured field extraction from invoices is Information Extraction, not Knowledge Assistant. E is wrong — batch Delta table LLM inference is `ai_query()`, not Multi-Agent Supervisor.

**Q41: A Generative AI Engineer is preparing a knowledge corpus for a RAG system that will include hundreds of scanned invoices (low-resolution PDFs), clean digital documents (high-quality PDFs), and Word files. During preprocessing, they must decide between high-accuracy OCR (slow, expensive) on all scanned images versus lower-accuracy, faster OCR with post-hoc correction. What is the primary trade-off to consider, and what best practice should guide the decision?**

   A. Always use high-accuracy OCR on everything to maximize final retrieval quality, regardless of cost
   B. Use only low-accuracy OCR to minimize preprocessing cost; retrieval quality is determined by the embedding model alone, not by OCR fidelity
   C. Apply OCR fidelity proportional to document value: high-accuracy OCR for critical revenue-bearing documents, lower-accuracy with human-in-the-loop correction for mid-tier documents, skip OCR on born-digital documents
   D. Store raw images in the vector index instead of converting to text; the embedding model will handle visual understanding automatically

A: **C**. The trade-off between OCR accuracy and cost must be risk-weighted: high-stakes documents (contracts, invoices driving revenue) justify expensive, high-accuracy OCR; mid-tier documents benefit from faster OCR plus targeted human review; born-digital documents (PDFs with embedded text) should never be re-OCR'd. This proportional investment maximizes quality per dollar spent and aligns preprocessing effort with business impact. **A is wrong because** high-accuracy OCR on low-value documents wastes cost without proportional benefit; this is over-engineering. **B is wrong because** OCR quality directly affects retrieval quality—garbled text cannot be recovered by the embedding model during inference, so low-accuracy OCR creates a permanent quality floor in the knowledge base. **D is wrong because** current embedding and retrieval models operate on text, not raw images; storing images wastes storage and retrieval index space, and vector search operates on text embeddings, not visual features.

**Q42: A Generative AI Engineer has built a LangChain LCEL chain with a stateful tool that tracks conversation context. The chain structure is: LLM → (if tool_call) → Tool → Store state in-memory → Return to LLM. During testing with multiple sequential requests from the same user, the engineer observes that state from Request 2 leaks into Request 3, even though each request should have isolated state. What is the likely root cause, and how should the chain be redesigned?**

   A. The LLM model is non-deterministic; use a lower temperature to fix state leakage
   B. The in-memory state dictionary is declared at module level (outside the chain invocation), causing all requests to share the same mutable object; state should be passed explicitly through the chain context or stored in a request-scoped datastore (e.g., session table in Delta)
   C. The tool is calling an external API; external APIs always leak state between requests
   D. State leakage is expected behavior in LCEL chains and cannot be avoided; the engineer should redesign the entire application without stateful tools

A: **B**. A module-level (global) mutable dictionary is shared across all invocations—each request appends to the same object instead of getting a fresh state. The fix is to either (1) instantiate a new state dict at the beginning of each chain invocation (request-scoped), (2) pass state as an explicit input through `RunnableParallel` or `RunnablePassthrough`, or (3) externalize state to a session-backed Delta table or cache so each request queries its own isolated session. **A is wrong because** model temperature controls output randomness, not state object identity—non-deterministic output is unrelated to shared mutable state. **C is wrong because** external APIs do not inherently cause state leakage; the leakage occurs in the Python runtime where the shared dict lives, not in remote services. **D is wrong because** state leakage in LCEL is avoidable with proper scoping; stateful tools are legitimate and widely used; the problem is poor state lifecycle management, not an inherent design flaw.

**Q43: A Generative AI Engineer wants their custom agent to call external tools using a standardized, interoperable protocol rather than writing bespoke integration code for each tool. Which Databricks feature supports this?**

O: a) MCP (Model Context Protocol) servers
   b) Delta Sharing
   c) Lakehouse Monitoring
   d) Inference logging with MLflow

A: **A**. MCP (Model Context Protocol) servers provide a standardized protocol for agents to discover and call tools without writing integration code for each tool. This interoperability is MCP's core value proposition: whether the tool is a managed Databricks feature, an external API, or a custom internal service, MCP provides a uniform discovery and invocation interface. **B is wrong because** Delta Sharing is for data access and lineage, not for agent tool integration. **C is wrong because** Lakehouse Monitoring is for data quality and schema drift detection, not for standardized tool protocols. **D is wrong because** inference logging captures request/response traces for observability, not for standardized tool protocol integration.

**Q44: A Generative AI Engineer has built a custom MCP server that exposes an internal inventory API as a tool. During a code review, a colleague asks how the agent discovers and invokes tools on the MCP server. Which statement BEST describes how tool discovery works with MCP?**

O: a) The agent hardcodes the tool name and parameters at build time; no runtime discovery is possible
   b) The MCP server exposes a tool manifest that the agent queries at runtime to discover available tools, their names, descriptions, and input schemas — allowing the agent to select and call the right tool without hardcoded integration logic
   c) Tools are only discoverable by reading the MCP server's source code
   d) The agent must be retrained every time a new tool is added to the MCP server

A: **B**. A core feature of the Model Context Protocol is runtime tool discovery: the MCP server exposes a structured manifest of available tools (names, descriptions, input schemas). The agent queries this manifest dynamically, which means new tools can be added to the server without changing agent code — a key advantage over hardcoded integrations. Option A describes the old hardcoded approach that MCP is designed to replace. Options C and D are incorrect — no source code reading or retraining is required.

**Q47: A Generative AI Engineer wants to iterate on a prompt in a dev environment, then promote the exact validated version to production without manually copy-pasting text between notebooks. Which Databricks feature supports this workflow?**

O: a) MLflow Prompt Registry (versioning and aliasing prompts across environments)
   b) A shared Word document
   c) Git commit messages only
   d) Delta table row filters

A: **A**. The MLflow Prompt Registry supports versioning, aliasing, and promoting prompts across environments (e.g., dev to production), avoiding manual copy-paste and preserving an audit trail. This workflow is essential for governance: engineers can validate a prompt in dev, register it as version v2, test it against QA datasets, then move the `@production` alias from v1 to v2 atomically when ready to promote. **B is wrong because** a shared Word document has no version control, audit trail, or integration with MLflow/Databricks infrastructure—it breaks reproducibility and makes rollback nearly impossible. **C is wrong because** git commit messages are code versioning only; they don't provide a structured registry for prompt artifacts or cross-environment promotion workflows that MLflow Prompt Registry handles. **D is wrong because** Delta table row filters are access control mechanisms for data, not artifact versioning or promotion—they have no relevance to prompt management.

   c) Email the model weights to each user.
   d) Disable the serving endpoint to reduce cost.

This abstracts away the serving endpoint's technical details and provides a user-friendly chat interface within a tool the user already uses. **A is wrong because** requiring users to write raw Python creates massive adoption friction and is not scalable—most end users do not have Python skills and should not need to learn them just to interact with an agent. **C is wrong because** emailing model weights is nonsensical; model weights are binary artifacts with no value to end users, and they would not know how to use them. **D is wrong because** disabling the serving endpoint entirely means the agent is not accessible at all, defeating the entire purpose of deployment.

**Q52: A Generative AI Engineer needs to run an LLM prompt across millions of rows in a Delta table as a batch job, rather than serving real-time requests. Which Databricks SQL function is designed for this?**

O: a) ai_query()
   b) VECTOR_SEARCH() used purely for row counts
   c) CREATE MODEL SERVING ENDPOINT
   d) DESCRIBE HISTORY

A: **A**. `ai_query()` is a Databricks SQL built-in function that applies an LLM call to each row of a query result, making it the appropriate choice for batch, non-interactive inference within SQL or notebook workloads — far preferable to looping row-by-row against a real-time serving endpoint. For very large tables (hundreds of millions of rows), additional Spark-level parallelism or incremental processing strategies may be needed alongside `ai_query()`. `VECTOR_SEARCH()` (B) is for similarity search, not LLM inference. `CREATE MODEL SERVING ENDPOINT` (C) and `DESCRIBE HISTORY` (D) are unrelated to batch inference.

**Q54: A Generative AI Engineer has enabled hybrid search on a Databricks AI Search (Vector Search) index. At query time, they want to weight the keyword score and the vector score differently depending on query type — more keyword weight for exact product code lookups, more vector weight for open-ended questions. Where is this weighting controlled?**

O: a) It must be baked into the embedding model weights during training
   b) At query time, using the `query_type` and score-weighting parameters available in the Vector Search / AI Search query API
   c) Only in the index creation configuration; it cannot be changed per query
   d) Hybrid search always uses a fixed 50/50 keyword-to-vector weighting with no adjustment possible

A: **B**. Databricks AI Search (Vector Search) hybrid search exposes query-time parameters that allow the caller to control the relative weighting of keyword (BM25) and vector similarity scores. This means the same index can serve both exact-identifier queries (high keyword weight) and semantic queries (high vector weight) without rebuilding the index. Options C and D incorrectly claim the weighting is fixed at index creation or at a constant 50/50 ratio. Option A is incorrect — embedding model weights are irrelevant to BM25/vector score blending.
