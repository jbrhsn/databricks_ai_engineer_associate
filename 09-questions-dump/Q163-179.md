**Q163: Which TWO scenarios correctly indicate that RAG is a BETTER choice than fine-tuning for a generative AI application?**

   A. The knowledge base is updated frequently and the application must always reflect the latest information.
   B. Full fine-tuning always produces more accurate outputs than RAG for the same domain.
   C. The application must be able to cite specific source documents for every answer.
   D. RAG is the better choice when the desired output style must closely match a proprietary writing voice.
   E. RAG is the better choice when the model needs to learn a new reasoning skill not present in pretrained weights.

A: **A and C**. **A qualifies** because RAG retrieves from a live, updatable knowledge base at inference time — when the knowledge base changes, the application immediately reflects those changes without any model retraining. Fine-tuning bakes knowledge into model weights, so every update requires a new training run. **C qualifies** because the retrieval step in RAG returns the actual source chunks used to construct the answer, giving the application traceable, document-level provenance. Fine-tuning internalises knowledge into weights, making it impossible to reliably attribute any single output to a specific source document. **B is wrong** because fine-tuning does not universally outperform RAG for the same domain — RAG consistently beats fine-tuning for factual recall tasks, especially on recent or frequently changing information. **D is wrong** because adapting output style (tone, register, writing voice) is precisely what fine-tuning excels at; RAG does not alter how the model writes, only what information it has access to. **E is wrong** for the same reason as D: learning a new reasoning skill requires changing the model's weights through fine-tuning; retrieval cannot teach the model a new capability.

**Q164: Which TWO factors should weigh most heavily when selecting a foundation model for a production application with strict data residency requirements?**

   A. The model's GitHub star count and community popularity.
   B. Whether the model can be self-hosted or deployed within the organisation's own Databricks workspace so data never leaves the environment.
   C. Whether the model has the highest score on a public English-language benchmark leaderboard.
   D. Whether the model's license permits commercial use and self-hosting for the intended application.
   E. The physical colour scheme of the vendor's branding and logo.

A: **B and D**. **B qualifies** because data residency requirements mandate that sensitive data must not leave a defined geographic or organisational boundary. A model that can be self-hosted within the organisation's own Databricks workspace (e.g., via Databricks Model Serving with a downloaded open-weight model) ensures that input prompts and output responses never transit to a third-party API endpoint, satisfying residency obligations. **D qualifies** because self-hosting requires explicit license permission. Many foundation model licenses prohibit commercial use, restrict redistribution, or forbid hosting as a service — failing to verify this before deployment can create significant legal liability. A technically capable model is unusable if its license blocks the intended deployment mode. **A is wrong** because GitHub stars measure community interest, not data residency compliance or production readiness; a model with millions of stars may still operate exclusively via an external API. **C is wrong** because benchmark leaderboard rankings measure general capability, not compliance properties; the highest-scoring model on an English leaderboard may not be self-hostable and therefore cannot satisfy residency requirements regardless of its performance. **E is wrong** because vendor branding has no bearing on technical compliance or legal licensing.

**Q165: A Generative AI Engineer is designing a chunking strategy for a large corpus of multi-section legal contracts. Which TWO practices are MOST likely to improve retrieval precision?**

   A. Chunk along natural clause and section boundaries so each chunk is a complete, self-contained legal unit.
   B. Split every contract into a single chunk to preserve full document context.
   C. Include parent-heading context (section title, clause number) as metadata on each chunk so it can be used for metadata filtering at retrieval time.
   D. Use fixed 100-token windows regardless of clause structure.
   E. Store all contracts concatenated in one row of a Delta table.

A: **A and C**. **A qualifies** because legal contracts have semantically meaningful structure — each clause or sub-clause covers a single obligation, right, or condition. Aligning chunk boundaries with those natural divisions ensures each chunk is coherent and self-contained, so that when a chunk is retrieved it fully answers the question rather than containing half a clause from two different topics. Fixed windows or arbitrary splits produce chunks that straddle boundaries, degrading both precision and comprehensibility. **C qualifies** because embedding-based similarity search alone cannot distinguish between clauses from different contract sections that use similar vocabulary. Attaching metadata (section title, clause number, contract party, date range) to each chunk enables metadata pre-filtering at retrieval time, narrowing the search space to relevant document regions before embedding comparison occurs. This directly reduces the number of irrelevant chunks returned. **B is wrong** because a single chunk per contract would mean every query retrieves the entire contract, making it impossible to surface the specific clause relevant to the query — precision collapses to near zero. **D is wrong** because 100-token fixed windows are agnostic to clause boundaries; a single clause may span hundreds of tokens while a short operative clause may be split mid-sentence, breaking meaning. **E is wrong** because concatenating all contracts into a single Delta row makes it impossible to chunk, index, or retrieve at the granularity of individual documents.

**Q166: Which TWO of the following are valid Databricks AI Search (Vector Search) index types and their defining characteristic?**

   A. Delta Sync index — automatically syncs embeddings with a source Delta table, with Databricks managing the embedding pipeline.
   B. Spark Streaming index — a continuously running Structured Streaming job that writes vectors to the vector store as new data arrives.
   C. Direct Vector Access index — allows the application to manage and upsert embeddings directly via the API, without a managed sync from a Delta table.
   D. JDBC index — connects to a relational database via JDBC and stores embeddings alongside relational rows.
   E. Static Snapshot index — a one-time bulk export that creates a fixed, never-updating vector index.

A: **A and C**. **A qualifies** because the Delta Sync index is one of the two natively supported index types in Databricks AI Search. It binds to a source Delta table and automatically keeps embeddings in sync as rows are added, updated, or deleted; Databricks handles embedding generation using a specified embedding endpoint, so the application does not need to manage the embedding pipeline. **C qualifies** because the Direct Vector Access index is the second supported index type. It exposes an upsert API that allows the application (or an external pipeline) to push pre-computed embedding vectors directly into the index, giving teams full control over the embedding model and update schedule. **B is wrong** because "Spark Streaming index" is not a named Databricks AI Search index type; Delta Sync covers streaming updates to source Delta tables automatically. **D is wrong** because JDBC connectivity is a Databricks data-ingestion feature unrelated to vector index storage types; no JDBC-based vector index type exists. **E is wrong** because no "Static Snapshot" index type is documented — the two supported types are Delta Sync and Direct Vector Access.

**Q167: After deploying a RAG application, the team observes that context_precision is 0.71 but context_recall is 0.93. Which TWO actions would MOST directly improve context precision without reducing recall below 0.85?**

   A. Reduce the embedding model's context window length.
   B. Add a re-ranking step to filter the initial broad retrieval down to the most relevant chunks before passing them to the LLM.
   C. Remove the system prompt entirely to eliminate any retrieval bias.
   D. Add metadata filtering (e.g., document type, date range) to the retrieval query so only relevant document subsets are searched, reducing irrelevant chunk retrieval.
   E. Increase the chunk size to reduce the total number of retrieved chunks.

A: **B and D**. **B qualifies** because a re-ranker (e.g., a cross-encoder model) operates as a post-retrieval filter: the initial retrieval casts a wide net (preserving the 0.93 recall) and the re-ranker then scores each retrieved chunk against the query and discards low-relevance ones before the context window is assembled. This directly raises precision — fewer irrelevant chunks reach the LLM — while the upstream broad retrieval protects recall. **D qualifies** because metadata filtering narrows the retrieval search space before embedding comparison occurs. If the query pertains only to contracts of a certain type or date range, filtering to that subset prevents unrelated documents from contributing low-precision chunks to the result set. Because the filter is applied as a pre-condition, recall within the relevant subset is maintained while precision improves. **A is wrong** because the embedding model's context window length affects how much text can be encoded per chunk, not the precision of the retrieval ranking — reducing it would not consistently remove irrelevant results. **C is wrong** because the system prompt governs the LLM's response behaviour, not the retrieval step; removing it would degrade output quality without affecting which chunks are retrieved. **E is wrong** because larger chunks mean fewer chunks are retrieved but each chunk covers more text, mixing relevant and irrelevant content — this can further reduce precision by diluting the signal within each chunk.

**Q168: A Generative AI Engineer is building a LangGraph agent. Which TWO statements correctly describe how state updates work in a LangGraph StateGraph?**

   A. Each node function receives the current state and returns a dict containing only the keys it wants to update — LangGraph merges this partial update into the full state.
   B. Nodes must return the complete state dict with all keys, even those they did not modify.
   C. For list-type state fields annotated with a reducer like `add_messages`, the returned value is appended to the existing list rather than replacing it.
   D. State is stored externally in a Databricks Delta table and nodes read and write it via SQL queries.
   E. Each node receives a copy of the state and must call `state.commit()` to persist changes back to the graph.

A: **A and C**. **A qualifies** because LangGraph's state management is designed around partial updates. A node only needs to return a dict containing the keys it modified; the framework merges those updates into the full state object using a shallow merge (or a reducer for annotated fields). This means nodes are loosely coupled — a retriever node that only updates `"context"` does not need to know about or return `"messages"`, `"tool_calls"`, or any other field. **C qualifies** because reducer annotations are the mechanism LangGraph uses to handle list accumulation. When a field is declared as `Annotated[list[BaseMessage], add_messages]`, the framework does not overwrite the field — it calls `add_messages(existing_list, new_messages)`, which appends new entries. Without the reducer, a plain `list` field would be overwritten on each update, losing history. **B is wrong** — returning the full state is unnecessary and incorrect; nodes return only the keys they own. **D is wrong** — LangGraph state lives in-process as a Python object (or in a checkpointer's storage such as a SQLite or Redis backend); it is not managed by Databricks Delta or accessed via SQL. **E is wrong** — there is no `state.commit()` method in LangGraph; the framework handles state persistence transparently through the checkpointer, if configured.

**Q169: Which TWO of the following correctly describe the role of a "tool" in a LangGraph or LangChain agent?**

   A. Tools must be written as Databricks SQL functions accessible only via `ai_query()`.
   B. A tool is a callable function decorated with `@tool` (or defined as a `BaseTool` subclass) that the LLM can choose to invoke by name, passing structured arguments, to perform an action or retrieve external information.
   C. A tool can only be invoked synchronously and cannot return structured data types such as dicts or Pydantic models.
   D. Tools must have a docstring or description that the LLM uses at runtime to decide whether to call the tool and what arguments to pass.
   E. Tools replace the need for a retriever in a RAG pipeline.

A: **B and D**. **B qualifies** because in both LangChain and LangGraph, a tool is fundamentally any Python callable that has been registered in the framework's tool interface — either via the `@tool` decorator on a plain function or by subclassing `BaseTool` for more complex cases. When the LLM generates a tool-call response (structured output naming the tool and arguments), the agent runtime invokes the callable and feeds the result back into the conversation. **D qualifies** because the LLM decides which tool to invoke — and how to construct its arguments — based entirely on the tool's name and description (docstring). Without a clear, accurate description, the LLM cannot reason about when and how to use the tool, leading to missed invocations or malformed arguments. The description is not optional metadata; it is the interface contract between the model and the tool. **A is wrong** because tools can be any Python function — they are not restricted to Databricks SQL functions or `ai_query()`. SQL functions registered in Unity Catalog can be exposed as tools, but that is one option, not a requirement. **C is wrong** because tools can and frequently do return structured data: a search tool may return a list of `Document` objects, and a database tool may return a dict or a Pydantic model. **E is wrong** because tools and retrievers are complementary — a retriever is typically wrapped as a tool in agentic RAG, but neither replaces the other architecturally.

**Q170: A Generative AI Engineer is writing a `mlflow.pyfunc.PythonModel` subclass to wrap a RAG chain. Which TWO statements about the `predict` method are correct?**

   A. The method signature must be `predict(self, context, model_input)` — where `context` is an `mlflow.pyfunc.PythonModelContext` object providing access to logged artifacts.
   B. The method must be named `run` for compatibility with Databricks Model Serving.
   C. The method should handle both dict-format and DataFrame-format inputs gracefully if the model will be called from multiple clients.
   D. `context` is a Python `contextvars.Context` object from the standard library.
   E. The method must return a Spark DataFrame for compatibility with Databricks Model Serving.

A: **A and C**. **A qualifies** because `predict(self, context, model_input)` is the mandated signature for any `mlflow.pyfunc.PythonModel` subclass. The `context` argument is an `mlflow.pyfunc.PythonModelContext` instance — not a standard library object — which exposes `context.artifacts` (a dict of local paths to files logged with `log_model`) and `context.model_config`. This is the only supported way to access logged artifacts at inference time. **C qualifies** because MLflow model serving may send inputs as a pandas DataFrame (from REST clients following the MLflow scoring protocol) or as a Python dict (from Python `mlflow.pyfunc.load_model().predict()` calls). A robust `predict` implementation checks the type of `model_input` and handles both cases, improving portability across deployment targets. **B is wrong** — the method must be named `predict`, not `run`; using any other name means the framework will not dispatch calls to it. **D is wrong** — `context` is an MLflow-specific `PythonModelContext` object, not `contextvars.Context` from the Python standard library; confusing these two is a common mistake for engineers who have used Python's async context variables. **E is wrong** — `predict` must return a pandas DataFrame, a list, a dict, or a numpy array; a Spark DataFrame is not a valid return type for a PyFunc model endpoint, which operates in a non-Spark serving container.

**Q171: Which TWO of the following prompt engineering techniques are MOST effective for eliciting structured JSON output from an LLM reliably?**

   A. Explicitly specify the exact JSON schema in the prompt, naming every required key and its expected type.
   B. Increase the temperature to 1.5 so the model explores more format variations before settling on JSON.
   C. Provide a one-shot or few-shot example of a correctly formatted JSON response in the prompt so the model has a concrete pattern to follow.
   D. Instruct the model to "think freely" without format constraints and post-process the output with regex.
   E. Chain-of-thought reasoning always produces valid JSON automatically, without additional format instructions.

A: **A and C**. **A qualifies** because explicitly declaring the JSON schema in the prompt removes ambiguity about what the model should produce. When the model is told "respond with a JSON object containing exactly these keys: `{\"intent\": string, \"confidence\": float, \"entities\": list}`", it has a precise contract to satisfy. Schema-in-prompt is especially important for models that do not support native structured output (e.g., function calling / response format enforcement), and it reinforces the format even for models that do. **C qualifies** because few-shot examples are among the highest-leverage prompt engineering techniques for format compliance. A concrete example demonstrates the exact spacing, quoting style, nesting, and field naming the model should replicate — it is easier for the model to copy a pattern than to generate one from a natural-language description alone. One well-chosen example often outperforms three paragraphs of format instructions. **B is wrong** because raising temperature increases output entropy — at 1.5 the model is more likely to generate creative but malformed JSON, not more likely to converge on a correct structure. For structured output, temperature should be lowered (0.0–0.3), not raised. **D is wrong** because unconstrained generation followed by regex post-processing is fragile: the model may produce prose, partial JSON, or inconsistent nesting that no regex reliably captures; this approach breaks under any slight variation in phrasing. **E is wrong** because chain-of-thought reasoning produces verbose reasoning traces in natural language — it does not automatically produce JSON. CoT and structured output are orthogonal techniques; combining them requires explicit format instructions in addition to the reasoning prompt.

**Q172: A Generative AI Engineer is building a multi-turn chat agent. Which TWO mechanisms are appropriate for maintaining conversation context across multiple turns in a LangGraph agent?**

   A. Calling `llm.reset()` between each turn to clear context and start fresh.
   B. In-state message history using the `add_messages` reducer on a `messages` field in the LangGraph state — each turn appends new messages, giving the LLM access to full conversation history.
   C. Storing conversation history only in a browser cookie with no server-side persistence.
   D. A LangGraph checkpointer (e.g., `MemorySaver` or a persistent store) that persists the agent's state between invocations so conversation history survives across separate API calls.
   E. Using a larger embedding model to encode the full conversation as a single vector passed to the LLM.

A: **B and D**. **B qualifies** because the `add_messages` reducer on a `messages: Annotated[list, add_messages]` state field is the idiomatic LangGraph pattern for multi-turn context within a single session. Every node that generates a response appends its message to the list; the next LLM call receives the full accumulated message history as its input, giving the model access to everything said in prior turns. **D qualifies** because a checkpointer decouples state persistence from the in-process execution lifetime. With a `MemorySaver` (for testing) or a database-backed checkpointer (for production), the agent's complete state — including the `messages` list — is written to durable storage after each invocation and reloaded at the start of the next. This is what makes conversation history survive across separate HTTP requests or agent restarts, which is essential for production deployments. **A is wrong** — there is no `llm.reset()` method in LangChain/LangGraph; even if such a method existed, resetting between turns would destroy conversation history rather than preserve it. **C is wrong** because a browser cookie stores data client-side only; the server-side LangGraph agent has no access to cookie data unless the client explicitly passes it in each request, which is not a LangGraph-native mechanism and creates security risks. **E is wrong** because LLMs process sequences of tokens, not embedding vectors of prior conversations; encoding history into an embedding vector discards the token-level detail the model needs to reason about prior turns.

A: **A and C**. **A qualifies** because `databricks.agents.deploy()` creates or updates a Model Serving endpoint for the registered agent and automatically enables inference tables, which log every request payload and response to a Delta table.    A. Use `mlflow.langchain.log_model()` when the chain is built from LangChain or LangGraph components — it handles LangChain-specific serialisation, dependency capture, and provides the native LangChain model flavour for serving.
   B. `mlflow.langchain.log_model()` always produces a smaller model artifact than `mlflow.pyfunc.log_model()`.

**Q175: A Generative AI Engineer wants to promote a registered Unity Catalog model version from dev to production. Which TWO mechanisms does Unity Catalog's MLflow Model Registry support for this workflow?**

   A. Physically moving the model artifact file between Unity Catalog schemas to signal promotion.
   B. Setting a named alias (e.g., `"champion"` or `"production"`) on a specific model version so that serving endpoints and downstream code can reference the alias rather than a hardcoded version number — enabling promotion by updating the alias.
   C. Using version-level tags to annotate the lifecycle stage (e.g., `{"stage": "production"}`) for tracking and governance purposes.
   D. Deleting all non-production versions to ensure only the current version is accessible.
   E. Calling `mlflow.transition_model_version_stage()` to set a "Production" stage on the Unity Catalog model version.

A: **B and C**. **B qualifies** because named aliases are the primary promotion mechanism in Unity Catalog's MLflow Model Registry. An alias such as `"champion"` is a mutable pointer that maps to one specific version number at any time. A CI/CD pipeline promotes a validated version by calling `MlflowClient().set_registered_model_alias(name, "champion", version)`, updating the pointer without changing any artifact. Serving endpoints and client code reference `models:/my_model@champion`, so they transparently pick up the new version when the alias is updated. **C qualifies** because Unity Catalog supports key-value tags on individual model versions. Tagging a version with `{"stage": "production", "approved_by": "mlops-team"}` provides an auditable, queryable governance record of which version is in which environment, complementing the alias-based serving mechanism. Tags do not affect serving resolution but are essential for lineage tracking and compliance. **A is wrong** — Unity Catalog model artifacts are immutable and location-locked; there is no mechanism to "move" an artifact file between schemas to signal promotion, and doing so would break artifact lineage. **D is wrong** — deleting non-production versions destroys reproducibility and audit history; Unity Catalog best practice is to retain all versions and use aliases to route traffic. **E is wrong** — `mlflow.transition_model_version_stage()` and stage concepts like "Staging"/"Production" belong to the legacy Workspace Model Registry API, which predates Unity Catalog. The Unity Catalog registry does not support stages; aliases are the replacement mechanism.

**Q176: Which TWO of the following are valid approaches for CI/CD of a Databricks agent prompt when using the MLflow Prompt Registry?**

   A. Store prompt versions in the Prompt Registry with environment-specific aliases (e.g., `"dev"`, `"staging"`, `"production"`); the CI/CD pipeline updates the alias to point to the validated version on promotion.
   B. Copy-paste the prompt text between notebooks manually at each promotion stage.
   C. Run automated evaluation (e.g., `mlflow.evaluate()` with quality judges) against the candidate prompt version in staging before the pipeline promotes the alias to `"production"`.
   D. Store the prompt only as a Python string constant in source code with no version tracking in the registry.
   E. Always deploy the most recently created prompt version without evaluation.

A: **A and C**. **A qualifies** because environment-specific aliases in the MLflow Prompt Registry provide the same alias-based promotion pattern used in the Model Registry. The prompt version in development carries the `"dev"` alias; after passing automated tests it is promoted by updating the `"staging"` alias, and finally the `"production"` alias. Application code loads the prompt by alias (`mlflow.load_prompt("prompts:/my_prompt@production")`), so the deployed behaviour changes the moment the alias is updated — no code deployment is required. **C qualifies** because automated quality evaluation is the gatekeeping step that makes prompt CI/CD meaningful. Running `mlflow.evaluate()` with LLM judges (e.g., groundedness, relevance, safety) against a fixed evaluation dataset on the candidate prompt version in staging provides an objective, reproducible quality gate. Only if the evaluation metrics meet the defined thresholds does the pipeline promote the alias to `"production"`. This prevents regressions from reaching users. **B is wrong** — manual copy-paste introduces human error, creates no audit trail, and cannot be automated into a pipeline; it defeats the purpose of a versioned registry. **D is wrong** — storing a prompt as a string constant in source code provides version control via Git but loses the semantic versioning, alias-based routing, and centralised governance that the Prompt Registry provides; it also couples prompt updates to code deployments. **E is wrong** — deploying the most recently created version without evaluation is reckless; prompt changes can degrade quality, introduce safety issues, or break downstream parsing — all of which automated evaluation would catch.

**Q177: Which TWO of the following correctly describe the defense-in-depth approach to preventing PII from appearing in LLM outputs in a Databricks RAG application?**

   A. Apply Unity Catalog column masking policies on PII fields in the source Delta table so that PII is masked or redacted before it can be retrieved and injected into LLM prompts.
   B. Rely solely on the LLM's safety training to avoid mentioning PII — no additional controls are needed.
   C. Configure AI Gateway output filtering to detect and redact PII patterns in generated responses before they are returned to the caller.
   D. Use a larger embedding model to dilute the PII signal in chunk vectors.
   E. Increase the temperature so the LLM is less likely to reproduce exact PII strings.

A: **A and C**. **A qualifies** because Unity Catalog column masking is a data-layer control that prevents PII from ever entering the retrieval pipeline. By applying a masking policy (e.g., replacing SSNs with `"XXX-XX-XXXX"` or nulling email addresses) on the source Delta table columns, the RAG retriever fetches already-masked text. Since the LLM prompt is constructed from retrieved chunks, PII never reaches the prompt context — it cannot appear in the output because it was never in the input. This is the earliest and most robust layer of the defense. **C qualifies** because AI Gateway output filtering provides a second, independent layer of protection that catches any PII that might slip through the data layer — for example, PII in metadata fields not covered by masking, or PII generated by the model itself from memorised training data. AI Gateway sits between the LLM response and the caller; it can apply regex-based or model-based PII detection and redact matches before the response is returned. **B is wrong** because LLM safety training is probabilistic and not auditable — it cannot be relied upon as a sole control in a compliance context. Safety fine-tuning reduces the likelihood of PII reproduction but does not guarantee prevention, especially when PII appears verbatim in the retrieved context. **D is wrong** because the embedding model converts text to dense vectors for similarity search; it does not modify the original text stored in chunks. The retrieved chunk text, which feeds the LLM prompt, is unchanged regardless of embedding model size. **E is wrong** because temperature controls output randomness, not information suppression. Higher temperature makes exact reproduction of long strings slightly less likely but also degrades coherence and factual accuracy — it is not a PII control and would be rejected by any security review.

**Q178: A Generative AI Engineer wants to run continuous evaluation on live production traffic without a ground-truth dataset. Which TWO Databricks built-in MLflow judges can operate in this reference-free mode?**

   A. `answer_correctness` — assesses whether the generated answer is factually correct.
   B. `groundedness` — assesses whether the generated answer is supported by the retrieved context.
   C. `chunk_relevance` — assesses whether retrieved chunks are relevant to the question.
   D. `exact_match` — assesses whether the model output matches a reference answer character for character.
   E. `perplexity` — measures the language model's uncertainty over the generated token sequence.

A: **B and C**. **B qualifies** because `groundedness` is a reference-free judge: it requires only the question, the retrieved context chunks, and the generated answer. It asks whether every claim in the answer is supported by the provided context — no ground-truth reference answer is needed. This makes it directly applicable to production traffic where labels do not exist, and it catches the critical failure mode of hallucination (claims not grounded in retrieved content). **C qualifies** because `chunk_relevance` is also reference-free: it requires only the question and the retrieved chunks. It scores whether each retrieved chunk is actually relevant to the question, surfacing retrieval failures (e.g., the retriever returning topically unrelated documents) without needing a labelled dataset. Both judges are included in the Databricks Agent Evaluation framework specifically to enable ongoing production monitoring. **A is wrong** because `answer_correctness` measures factual accuracy against a known ground-truth reference answer — without one, the judge has nothing to compare against and cannot be run in reference-free mode. **D is wrong** because `exact_match` is a string-comparison metric that requires a reference string; it is fundamentally reference-dependent and cannot evaluate production traffic without labelled examples. **E is wrong** because `perplexity` is not a Databricks built-in MLflow judge in the Agent Evaluation framework; it is a language modelling metric defined at the model level, not a response-quality judge applicable to RAG evaluation.

**Q179: Which TWO of the following correctly describe the purpose and usage of MLflow Tracing for a deployed LangGraph agent?**

   A. MLflow Tracing captures a hierarchical record of all spans in an agent invocation — including LLM calls, tool calls, and retriever calls — with their inputs, outputs, and latencies, enabling root-cause analysis of production failures.
   B. MLflow Tracing replaces the need for inference tables — once tracing is enabled, inference tables can be disabled.
   C. Traces are stored only in-memory and are lost when the serving endpoint restarts.
   D. With `mlflow.langchain.autolog()` enabled, LangGraph agent invocations are automatically traced without requiring manual span instrumentation in every node function.
   E. MLflow Tracing only captures the final LLM output, not intermediate tool calls or retriever results.

A: **A and D**. **A qualifies** because the defining value of MLflow Tracing is its hierarchical span structure. A single agent invocation produces a tree of spans: the root span covers the full request, child spans cover each LLM call, grandchild spans cover each tool call or retriever lookup, and so on. Each span records its start time, end time, inputs, and outputs. This level of granularity allows an engineer to pinpoint whether a latency spike came from the retriever, a slow tool API, or an LLM cold start — something that flat inference table logs cannot provide. **D qualifies** because `mlflow.langchain.autolog()` is a one-line setup call that instruments all LangChain and LangGraph components automatically. When autologging is active, every LLM call, chain invocation, tool execution, and retriever call within a LangGraph graph is wrapped in a span without any per-node manual instrumentation. This makes it practical to enable comprehensive tracing in production with minimal code changes. **B is wrong** because MLflow Tracing and inference tables serve complementary but distinct purposes: inference tables capture request/response pairs for evaluation and auditing, while traces capture the internal execution hierarchy for debugging. Disabling inference tables would eliminate the evaluation data needed for continuous monitoring — tracing does not replace that function. **C is wrong** because MLflow Traces are persisted to the MLflow backend store (in Databricks, the Unity Catalog-backed MLflow tracking server) — they are durable across restarts and queryable via the MLflow UI and API. **E is wrong** because capturing only the final LLM output is the opposite of what MLflow Tracing does; its primary value is precisely the visibility into intermediate steps — tool invocations, retriever results, and multi-hop LLM calls — that are otherwise opaque.
