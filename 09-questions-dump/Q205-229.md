**Q205: A LangGraph agent uses an `InMemorySaver` checkpointer. After restarting the Python process, the application invokes the graph with the same `thread_id` used in the previous session. What behavior results, and why?**

   A. The graph resumes from the previous state, including all prior messages and state fields, because `thread_id` is sufficient to restore memory across process restarts.
   B. The graph starts a fresh conversation with no prior messages, because `InMemorySaver` stores checkpoints in RAM, which is cleared on process restart. The same `thread_id` in a new process resolves to an empty checkpoint store.
   C. The graph throws a `CheckpointerNotFoundError` because the checkpoint data no longer exists.
   D. The graph automatically falls back to `SqliteSaver` to recover the lost state.

A: **B**. `InMemorySaver` persists checkpoints only in process memory. On restart, the memory is wiped — the thread's checkpoint data is gone, even if a new `thread_id` is reused. The `thread_id` is a lookup key, and without a checkpoint to retrieve, the graph initialises fresh. This is a common misconception: developers assume the `thread_id` alone is sufficient for stateful multi-turn, not realizing the backend storage is critical. Options A and D misunderstand the memory model; option C is close but the error is silent (no exception is raised — the graph just starts fresh).

**Q206: Which TWO of the following statements correctly describe conditional edges in LangGraph and their role in implementing agentic loops?**

   A. Conditional edges always execute in parallel with static edges, causing both to fire on every super-step.
   B. A conditional edge calls a router function that inspects the current state and returns the name of the next node to visit at runtime.
   C. To implement a loop where the agent calls tools and returns to the agent node, use `add_conditional_edges("agent", router_fn, {...})` to route to tools when tool calls are present, and add a separate static edge from tools back to agent.
   D. The router function in `add_conditional_edges` must return a list of node names, never a single name or `END`.
   E. Conditional edges are only for branching (multiple independent paths); they cannot create loops back to earlier nodes.

A: **B and C**. B is correct because conditional edges are exactly routing functions that decide the next node based on state inspection. C is correct because the full pattern for an agent-tools loop is: conditional edge on agent determines next destination (tools or END), and a static edge from tools back to agent ensures the loop closes. A is wrong because conditional and static edges don't run in parallel; the graph executes one edge at a time based on the current node. D is wrong because router functions can return single strings or lists (for parallel execution) or `END` — flexibility is the point. E is wrong because loops are created by routing back to earlier nodes.

**Q207: A LangGraph agent has a state reducer `Annotated[list[str], operator.add]` on the `messages` field. Two nodes run in parallel during the same super-step: Node A returns `{"messages": ["msg_A"]}` and Node B returns `{"messages": ["msg_B"]}`. What is the final state of the `messages` field after the super-step completes?**

   A. `["msg_A"]` only, because Node A runs first alphabetically.
   B. `["msg_B"]` only, because the second node's update overwrites the first.
   C. `["msg_A", "msg_B"]` because the `operator.add` reducer concatenates both updates into the existing list.
   D. A conflict error is raised because two nodes updated the same field simultaneously.

A: **C**. The `operator.add` reducer is the whole point of `Annotated` — it enables safe parallel node execution by concatenating updates rather than overwriting. Both nodes return their updates; the reducer applies `add` to merge them: if the prior state was `[]`, both updates merge to `["msg_A", "msg_B"]`. Without the reducer (default overwrite), the answer would be C or D depending on execution order — reducers are how LangGraph makes parallelism deterministic. Options A and B reflect a misunderstanding of reducers and parallel execution.

**Q210: An agent developer wraps an `RunnableLambda` around a custom Python function that formats retrieved documents into a context string. This function takes a `List[Document]` and returns a `str`. In an LCEL chain, where retriever output flows into the lambda, what guarantees does `RunnableLambda` provide that a raw Python function does not?**

A: **C**. The context window is the total budget (input + output combined); `max_tokens` caps the *output* portion. If the input were 4,500 and max_tokens were 3,000, then total = 7,500 > 6,144, and truncation or an error would occur. Option B wrongly conflates max_tokens with window size; option D is wrong because there's room without truncation; option A is right on outcome but doesn't explain the mechanism.
