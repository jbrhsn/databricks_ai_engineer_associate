**Q20: A Generative AI Engineer needs to reduce hallucinations in a RAG application where the LLM sometimes answers using its own parametric knowledge instead of the retrieved context. Which approach would BEST address this?**

O: a) Increase the temperature parameter to encourage more creative responses.
   b) Add explicit instructions in the system prompt telling the LLM to only answer using the provided context and to say "I don't know" if the answer isn't present.
   c) Switch to a larger LLM with more parameters.
   d) Remove the retrieved context entirely and rely on the LLM's built-in knowledge.

A: **B**. Explicitly grounding instructions in the system prompt constrains the LLM to use only the retrieved context, which directly addresses hallucinations by preventing the model from relying on its parametric knowledge. When the instruction is clear ("answer only using the provided context") and paired with an explicit "I don't know" fallback, the LLM learns to recognize when information is absent and refrains from inventing answers. **A is wrong because** increasing temperature makes the LLM *more* creative and less constrained—it would amplify hallucination, not reduce it, because higher temperature encourages the model to deviate from its instructions and follow its own learned patterns. **C is wrong because** a larger LLM with more parameters actually tends to hallucinate *more* confidently (it has more parametric knowledge to draw from) rather than less; model size alone does not reduce hallucination without proper grounding constraints. **D is wrong because** removing the retrieved context entirely eliminates the mechanism for grounding answers and guarantees hallucination—the LLM has nothing but its own knowledge, which is the problem being addressed.

**Q21: A Generative AI Engineer wants to reduce the cost and latency of their RAG application without significantly sacrificing answer quality. Which technique would help MOST?**

O: a) Increase the number of chunks retrieved per query.
   b) Implement a re-ranker to select only the most relevant chunks before passing them to the LLM, reducing the number of tokens sent.
   c) Switch to a larger context window model so all chunks can be included.
   d) Remove the vector database and pass the full document corpus to the LLM each time.

A: **B**. A re-ranker filters retrieved chunks down to the most relevant ones, reducing the token count sent to the LLM (lowering cost and latency) while preserving answer quality.

**Q22: Which Databricks feature would be MOST appropriate for storing and serving embeddings for low-latency similarity search in a production RAG application?**

O: a) Delta Live Tables
   b) Databricks Vector Search
   c) Unity Catalog Volumes
   d) Databricks Feature Store for tabular features only

A: **B**. Databricks Vector Search is purpose-built for storing embeddings and performing efficient low-latency similarity search, which is essential for production RAG retrieval operations. It is optimized specifically for vector data with configurable index types and provides sub-100ms query latency at scale, far superior to generic storage approaches. **A is wrong because** Delta Live Tables are designed for ETL pipeline orchestration and data freshness, not for vector indexing or similarity search—they store traditional tabular data, not embeddings. **C is wrong because** Unity Catalog Volumes are file storage and do not provide indexing or search capabilities; storing embeddings as files without an index would require manual parsing and comparison on every query, resulting in unacceptable latency for production. **D is wrong because** the Databricks Feature Store is designed for tabular machine learning features and does not support vector embeddings or similarity search operations required for RAG retrieval.

**Q23: A Generative AI Engineer is fine-tuning an open-source LLM for a specific domain but has a very limited labeled dataset (a few hundred examples). Which approach is MOST appropriate?**

O: a) Full fine-tuning of all model parameters.
   b) Train the model from scratch on the domain data.
   c) Use parameter-efficient fine-tuning (e.g., LoRA) to adapt the model with fewer trainable parameters.
   d) Ignore fine-tuning and use prompt engineering exclusively with no other adjustments possible.

A: **C**. Parameter-efficient fine-tuning methods like LoRA require far less data and compute than full fine-tuning, making them well-suited for small labeled datasets while still allowing meaningful domain adaptation.

**Q24: A Generative AI Engineer wants to ensure that a customer-facing chatbot does not leak personally identifiable information (PII) in its responses. Which approach is MOST appropriate?**

O: a) Rely solely on the LLM's internal safety training to avoid PII leakage.

A: **B**. A dedicated guardrail layer for PII detection and redaction on both inputs and outputs provides a reliable, auditable safety mechanism rather than relying on implicit model behavior.

O: a) The number of parameters in the model.
   b) Whether the model supports multiple languages.

Parameter count (A) and language support (B) are secondary considerations. Licensing terms (D), while important for open-source adoption, are less deterministic than a hard compliance violation.

**Q26: Which MLflow component would a Generative AI Engineer use to track experiments, compare different prompt templates, and log evaluation metrics during LLM application development?**

O: a) MLflow Model Registry only
   b) MLflow Tracking
   c) MLflow Projects
   d) MLflow Recipes exclusively

A: **B**. MLflow Tracking is designed to log and compare parameters (like prompt templates), metrics, and artifacts across different experiment runs, making it the right tool for this iterative development process. Tracking enables side-by-side comparison of prompts against standard metrics like ROUGE, BERTScore, or custom LLM-judge scores, allowing engineers to identify which variations actually improve performance. **A is wrong because** the Model Registry is focused on versioning and deploying already-trained models, not on the experimental iteration and metric comparison phase that happens during development. **C is wrong because** MLflow Projects packages reproducible workflows but is not the component for tracking experimental runs and comparing metrics; it's more about encapsulation and deployment automation. **D is wrong because** MLflow Recipes are templates for common ML workflows (not specific to GenAI prompt experimentation) and do not provide the low-level tracking and comparison capabilities needed for this iterative development use case.

**Q27: A Generative AI Engineer built a RAG chatbot. Users report that the chatbot gives technically accurate answers but frequently fails to address what the user actually asked — it answers a related but different question. Which evaluation metric is BEST suited to surface this specific failure mode?**

O: a) Faithfulness/groundedness of the answer relative to the retrieved context
   b) Answer relevance — whether the generated answer actually addresses the user's question
   c) Perplexity of the generated response
   d) Context recall — whether all relevant documents were retrieved

A: **B**. Answer relevance measures whether the generated response directly addresses what the user asked, independent of whether the answer is factually grounded in context. Faithfulness/groundedness (A) measures whether the answer contradicts the retrieved context — it would not surface the problem of answering a related but different question. Perplexity (C) measures language model confidence, not question-answer alignment. Context recall (D) is a retrieval metric, not a generation quality metric.

**Q28: A Generative AI Engineer needs to give an LLM agent the ability to perform real-time calculations (e.g., mortgage payment calculations) that the LLM cannot reliably do on its own. What is the BEST approach?**

O: a) Fine-tune the LLM extensively on math problems.
   b) Provide the LLM with a calculator/code-execution tool it can call to perform the calculation and return the result.
   c) Increase the model's context window.

A: **B**. Giving the LLM access to a deterministic tool (like a calculator or code interpreter) for precise calculations is more reliable than relying on the LLM's own arithmetic reasoning, which is a common failure point for LLMs.

**Q30: A Generative AI Engineer notices their RAG application performs well on factual lookup questions but poorly on multi-hop reasoning questions that require combining information from multiple document sections. What is the MOST likely underlying issue?**

O: a) The embedding model is too large.
   b) The retrieval step only fetches a small number of chunks that individually may not contain enough combined context for multi-hop reasoning.
   c) The LLM's temperature is set too low.
   d) The vector store index is corrupted.

A: **B**. Multi-hop reasoning requires synthesizing information across multiple chunks; if retrieval returns too few or insufficiently diverse chunks, the LLM lacks the combined context needed to answer correctly.

**Q31: A Generative AI Engineer wants to A/B test two different prompt templates in production to determine which yields better user satisfaction. Which deployment strategy supports this?**

O: a) Deploy only one version and collect user feedback anecdotally over weeks until a clear winner emerges.
   b) Use a serving endpoint that supports traffic splitting between two model/prompt versions and collect metrics on each.
   c) Manually contact each user to ask which prompt they prefer before allowing them to interact with the system.
   d) Randomly degrade one of the two versions so that one will obviously perform worse.

A: **B**. A/B testing prompt versions in production requires routing a controlled percentage of live traffic to each version and collecting comparative metrics. On Databricks, this is achieved by configuring traffic routing at the AI Gateway layer (which supports weighted route splitting across endpoints) or via application-level routing logic — not as a native single-endpoint split. The key principle is that both versions run simultaneously on real traffic, enabling data-driven comparison rather than relying on anecdotal feedback (A), manual polling (C), or artificially degrading one option (D).

**Q35: When evaluating the retrieval component of a RAG system, which metric specifically measures whether the relevant document was retrieved, regardless of its rank position among the results?**

O: a) Recall
   b) Perplexity
   c) BLEU
   d) ROUGE

A: **A**. Recall measures whether relevant documents were successfully retrieved within the result set, which is a standard retrieval evaluation metric independent of exact ranking (as opposed to metrics like NDCG, which do weight rank position).

**Q39: A company wants to deploy an LLM-powered internal tool but needs to enforce rate limiting, authentication, and usage monitoring across multiple internal applications calling the same model endpoint. Which Databricks capability BEST supports all three of these requirements together?**

O: a) A shared Jupyter notebook with no access controls
   b) Databricks AI Gateway, which provides rate limiting (requests per minute per consumer), authentication, and usage tables for tracking consumption across calling applications
   c) Unity Catalog column masks, which control row-level data access
   d) Databricks Lakehouse Monitoring, which monitors data quality in Delta tables

A: **B**. Databricks AI Gateway is purpose-built for governing shared model endpoint access: it provides per-consumer rate limiting (RPM/TPM caps), authentication delegation, and usage tables that record token consumption per caller — satisfying all three stated requirements. Databricks Model Serving endpoints provide authentication and basic request logging but do not natively provide rate limiting. Unity Catalog column masks (C) govern data access, not model API access. Lakehouse Monitoring (D) monitors data pipelines, not LLM serving traffic.
