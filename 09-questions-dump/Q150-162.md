**Q150: In a LangGraph `StateGraph`, what correctly describes a "node" and how it is registered?**

O: a) A node is a Python class that inherits from `langgraph.Node` and is added by calling `graph.register_node(MyNodeClass)`.
   b) A node is a Python function (or callable) that accepts the current state and returns a dict of state updates; it is registered via `graph.add_node("name", function)`.
   c) A node is a YAML configuration block that describes the agent's step; it is loaded from a file and parsed by `StateGraph.from_yaml()`.
   d) A node is a Databricks notebook cell annotated with `# @node`, and the graph is built automatically from the notebook structure.

A: **b**. In LangGraph, a node is any Python callable — typically a function — that receives the current typed state dict and returns a partial state dict containing only the keys it wants to update. It is registered with `graph.add_node("node_name", my_function)`. The `StateGraph` is instantiated with a state schema (a `TypedDict` subclass or dataclass) so the framework knows which keys are valid. Option a is wrong because LangGraph does not require class inheritance from a base `Node` type — plain functions are the idiomatic choice. Option c is wrong because LangGraph has no YAML-based graph definition format; graphs are constructed in Python code. Option d is wrong because Databricks notebooks have no native `@node` annotation that auto-constructs a `StateGraph`.

**Q151: A LangGraph agent must route to a `"call_tool"` node when the LLM output contains a tool call, and to an `"end"` node otherwise. Which LangGraph construct implements this branching?**

O: a) `graph.add_edge("llm", "call_tool", condition=lambda state: state["tool_call"] is not None)` — a conditional `add_edge` with an inline lambda.
   b) `graph.add_conditional_edges("llm", routing_fn, {"tool": "call_tool", END: "end"})` — a routing function that returns a string key mapped to a target node.
   c) A Python `if/else` block placed inside the `"llm"` node function that directly calls the next node function before returning.
   d) Two separate `StateGraph` objects — one for the tool path and one for the direct-answer path — whose results are merged after execution.

A: **b**. `add_conditional_edges(source_node, routing_function, mapping)` is the LangGraph construct for conditional routing. The `routing_function` receives the current state and returns a string key; that key is looked up in the `mapping` dict to find the target node name. Using `END` (imported from `langgraph.graph`) as a key routes to graph termination. Option a is wrong because `add_edge` does not accept a `condition` parameter — it adds an unconditional edge; conditional logic requires `add_conditional_edges`. Option c is wrong because calling the next node function directly inside a node bypasses the graph's execution engine and breaks state management, checkpointing, and streaming. Option d is wrong because the use case requires a single graph with branching, not two independent graphs — LangGraph is designed to express these branches as conditional edges within one `StateGraph`.

**Q152: A LangGraph agent state schema includes the field `messages: Annotated[list, add_messages]`. When a node returns `{"messages": [new_ai_message]}`, what does the `add_messages` reducer do?**

O: a) It replaces the entire existing `messages` list with only the newly returned messages, discarding prior conversation history.
   b) It appends the new messages to the existing list, so conversation history accumulates across node invocations.
   c) It deduplicates messages across the list by comparing message content, keeping only unique entries.
   d) It converts all accumulated messages into a single concatenated string summary and stores that string in the `messages` field.

A: **b**. `add_messages` is a reducer function imported from `langgraph.graph.message`. When a node returns a partial state update, the `Annotated` type hint tells LangGraph to apply the `add_messages` reducer rather than perform a plain overwrite. The reducer appends incoming messages to the existing list, preserving the full conversation history as the graph traverses nodes. This is the standard pattern for building conversational agents in LangGraph. Option a describes a plain assignment (no reducer), which would be the behaviour if the field were declared as a bare `list` without an `Annotated` reducer. Option c is wrong because `add_messages` does not deduplicate by content; it simply appends. Option d is wrong because `add_messages` never collapses the list into a string — the messages field always remains a list of message objects.

**Q153: A Generative AI Engineer wraps a custom RAG pipeline as an MLflow PyFunc model. Which statement correctly describes the required class structure?**

O: a) The class must inherit from `torch.nn.Module` and implement a `forward(self, x)` method; MLflow's PyFunc flavor wraps `forward` automatically.
   b) The class must inherit from `mlflow.pyfunc.PythonModel` and implement a `predict(self, context, model_input)` method; `context.artifacts` provides access to logged artifacts and `model_input` carries inference-time input.
   c) The class can inherit from any Python base class as long as it implements a method named `run(self, input_data)` — MLflow detects the `run` method by convention.
   d) The class must inherit from `mlflow.pyfunc.PythonModel` and implement `load_context(self, context)` as the inference method; `predict` is optional.

A: **b**. An MLflow PyFunc custom model must inherit from `mlflow.pyfunc.PythonModel`. The required inference method is `predict(self, context, model_input)` — `context` is an `mlflow.pyfunc.PythonModelContext` object that exposes `context.artifacts` (a dict mapping artifact names to local file paths logged with the model) and `context.model_config`. `model_input` is the runtime input, typically a pandas DataFrame or a dict. Option a is wrong because `torch.nn.Module` with `forward` is the PyTorch flavour, not PyFunc; MLflow wraps those separately via `mlflow.pytorch`. Option c is wrong because the method name must be exactly `predict` — MLflow does not scan for arbitrary method names. Option d is wrong because `load_context` is an *optional* initialisation hook (used to load heavy state like model weights once at startup), not the inference method; `predict` is the required inference entry point.

**Q154: A Generative AI Engineer deploys a chat agent as a PyFunc model to Databricks Model Serving. What is the expected format of `model_input` in the `predict` method for a chat-compatible endpoint?**

O: a) A plain Python string containing only the user's most recent message, e.g. `"What is MLflow?"`.
   b) A pandas DataFrame with one row per token, where each row contains the token text and its position index.
   c) A dict with a `"messages"` key containing a list of message dicts, each with `"role"` (`"system"`, `"user"`, or `"assistant"`) and `"content"` keys — following the OpenAI chat completion format.
   d) A JSON array of raw text chunks retrieved from the vector store, passed directly without role metadata.

A: **c**. Databricks Model Serving passes requests to the `/invocations` endpoint. For chat-compatible endpoints (those conforming to the OpenAI chat completion standard), the payload is a dict with a `"messages"` key whose value is a list of message objects, each carrying `"role"` and `"content"`. This is the format the AI Playground, `mlflow.evaluate()` with `model_type="databricks-agent"`, and direct REST callers all use. Option a is wrong because a bare string has no role metadata, making it impossible for the model to distinguish system prompts from user turns in a multi-turn conversation. Option b is wrong because token-level DataFrames are not an MLflow chat serving convention — that format is specific to certain tokeniser utilities, not to a serving endpoint payload. Option d is wrong because raw retrieval chunks are internal data structures, not a defined serving input format; the endpoint receives the conversation, not the retrieved documents.

**Q155: Which TWO of the following are required when logging a RAG chain with MLflow so it can be registered to Unity Catalog and served correctly?**

   A. The model must be pre-trained on a GPU cluster before it can be logged to MLflow.
   B. A model signature (input/output schema) must be defined to specify the expected request and response format.
   C. The chain must first be exported as a Docker container image before calling `mlflow.langchain.log_model()`.
   D. The correct MLflow flavor (e.g., `mlflow.langchain.log_model()` or `mlflow.pyfunc.log_model()`) must be used, and all required Python dependencies must be specified.
   E. The Unity Catalog schema must be dropped and recreated to clear the model registry before registration.

A: **B and D**. A model signature (`mlflow.models.infer_signature()` or manually constructed `ModelSignature`) is required to define the input and output schema so that Model Serving can validate requests at runtime and so that the AI Playground can render the correct input form. Without a signature, serving endpoints may reject requests or behave unpredictably. The correct MLflow flavor is required because it determines how the model is serialised, how its environment is captured, and what MLflow can reconstruct at load time; using the wrong flavor or omitting the `pip_requirements` / `conda_env` means the model will fail to load in a fresh serving container. Option A is wrong because RAG chains are not "trained" in the classic sense — they assemble retrieval + generation at inference time and can be logged from any environment. Option C is wrong because `log_model` handles containerisation internally; the user never needs to build a Docker image manually before logging. Option E is wrong because dropping and recreating a UC schema is destructive and completely unnecessary — models are registered by calling `mlflow.register_model()` with the three-part UC name, and the schema must already exist but does not need to be recreated.

**Q156: A Generative AI Engineer wants to create a Delta Sync Vector Search index programmatically on Databricks. Which SDK call correctly creates the index?**

O: a) `mlflow.create_vector_index(endpoint_name=..., source_table=..., embedding_model=...)`
   b) `spark.sql("CREATE VECTOR INDEX my_index ON catalog.schema.table USING embedding_model=...")`
   c) `from databricks.vector_search.client import VectorSearchClient; client = VectorSearchClient(); client.create_delta_sync_index(endpoint_name=..., index_name=..., source_table_name=..., primary_key=..., embedding_source_column=..., embedding_model_endpoint_name=...)`
   d) `databricks.sdk.service.vectorsearch.VectorSearchIndexesAPI(w).create(name=..., primary_key=..., delta_sync_index_spec=...)`

A: **c**. The Databricks Vector Search Python SDK is imported from `databricks.vector_search.client`. `VectorSearchClient().create_delta_sync_index()` is the canonical high-level call that creates a managed index tied to a Delta source table: it provisions the index on the specified endpoint, configures automatic syncing from the Delta table, and sets up the embedding pipeline using the specified model endpoint. The key parameters are `endpoint_name` (the Vector Search endpoint to host the index), `index_name` (three-part UC name), `source_table_name`, `primary_key`, `embedding_source_column` (the text column to embed), and `embedding_model_endpoint_name`. Option a is wrong because `mlflow` has no `create_vector_index` API — MLflow is an experiment tracking and model management platform. Option b is wrong because Databricks SQL does not have a `CREATE VECTOR INDEX` DDL statement — vector index creation is an SDK/API operation, not a SQL DDL operation. Option d is wrong because the Databricks Workspace SDK (`databricks.sdk`) does expose a `VectorSearchIndexesAPI`, but it is a lower-level REST wrapper requiring a `VectorSearchIndexSpec` object — `VectorSearchClient` is the recommended high-level interface for this workflow.

**Q157: A Generative AI Engineer calls `databricks.agents.deploy(model_name, model_version)` after registering a LangGraph agent to Unity Catalog. What does this function do and what does it return?**

O: a) It registers the model to Unity Catalog — this is the step that creates the UC model entry; calling it again with a new version increments the version number. It returns the new model version number.
   b) It creates or updates a Model Serving endpoint configured for the agent, enables the Review App for human feedback collection, and activates inference table logging. It returns a deployment info object containing the endpoint URL and Review App URL.
   c) It sends the serialised model weights to an external API provider (e.g., OpenAI or Anthropic) for remote hosting and returns a provider-assigned deployment ID.
   d) It creates a Databricks Workflow (Jobs) that runs the agent on a scheduled basis and returns the workflow job ID.

A: **b**. `databricks.agents.deploy(model_name, model_version)` (from the `databricks-agents` SDK) is a one-call deployment primitive that: (1) creates or updates a Model Serving endpoint pointed at the specified UC model version, (2) enables the Databricks Review App — a web UI for SMEs and stakeholders to test the agent and submit per-response feedback — and (3) activates inference table logging so all requests and responses are captured to a Delta table for evaluation and monitoring. It returns a deployment info object with the endpoint name, endpoint URL, and Review App URL. Option a is wrong because UC model registration is a separate prior step done with `mlflow.register_model()` — `databricks.agents.deploy` is called *after* registration. Option c is wrong because Databricks agents are served on Databricks-managed infrastructure, not forwarded to external providers; external model access is configured separately via AI Gateway or external model endpoints. Option d is wrong because `databricks.agents.deploy` creates a *serving endpoint* (a real-time REST API), not a scheduled Workflow/Jobs run.

**Q158: A Generative AI Engineer wants to register a logged MLflow run's model artifact to Unity Catalog under `prod.agents.rag_v2`. Which `mlflow.register_model()` call is correct?**

O: a) `mlflow.register_model(model_uri="models:/prod/agents/rag_v2", name="rag_v2")`
   b) `mlflow.register_model(model_uri="runs:/<run_id>/model", name="prod.agents.rag_v2")`
   c) `mlflow.log_model(artifact_path="model", registered_model_name="rag_v2")` — UC registration is inferred automatically from the workspace default catalog.
   d) `spark.sql("REGISTER MODEL prod.agents.rag_v2 FROM RUN '<run_id>'")`

A: **b**. `mlflow.register_model(model_uri="runs:/<run_id>/model", name="prod.agents.rag_v2")` is the correct call. The `model_uri` uses the `runs:/` scheme to locate the artifact within a specific MLflow run, and `name` takes the full Unity Catalog three-part name (`catalog.schema.model_name`). Databricks must have `mlflow.set_registry_uri("databricks-uc")` configured (or it must be the workspace default) for the three-part name to be resolved against UC. Option a is wrong on two counts: the `models:/` URI scheme is used to *reference an already-registered version* (e.g., `models:/prod.agents.rag_v2/1`), not to pass a source artifact for registration; and passing only `"rag_v2"` as the name would register to the legacy Workspace Model Registry rather than UC. Option c is wrong because passing only the short name `"rag_v2"` to `registered_model_name` registers to the Workspace Model Registry — not to Unity Catalog — unless a fully qualified three-part name is used. Option d is wrong because Databricks SQL has no `REGISTER MODEL FROM RUN` DDL statement; model registration is an MLflow Python API operation.

**Q159: A Generative AI Engineer is logging a RAG chain to MLflow and wants to include a model signature derived from example data. Which code pattern correctly creates and attaches the signature?**

O: a) Call `signature = model.get_signature()` on the chain object itself before logging; MLflow reads the signature from the model object automatically.
   b) Construct the signature manually as `mlflow.models.ModelSignature(inputs=Schema([ColSpec("string")]), outputs=Schema([ColSpec("string")]))` and pass it to `log_model` — `infer_signature` is only for tabular ML models.
   c) `from mlflow.models import infer_signature; signature = infer_signature(model_input=example_input, model_output=example_output)` — then pass `signature=signature` to `mlflow.langchain.log_model()` or `mlflow.pyfunc.log_model()`.
   d) Signatures are inferred automatically at serving time; passing `signature=` to `log_model` raises a deprecation warning and should be omitted.

A: **c**. `mlflow.models.infer_signature(model_input, model_output)` is the standard utility that inspects example input and output data (dict, pandas DataFrame, or string) and constructs an `mlflow.models.ModelSignature` object encoding the input schema and output schema. This `signature` object is then passed as the `signature=` keyword argument to `mlflow.langchain.log_model()`, `mlflow.pyfunc.log_model()`, or any other `log_model` call. The signature is stored with the model artifact and used by Model Serving to validate incoming request payloads. Option a is wrong because LangChain chains and PyFunc models do not have a `get_signature()` method — signatures are always created externally via `infer_signature` or manual construction. Option b is wrong that `infer_signature` is restricted to tabular ML — it works with any Python objects including dicts and strings, making it fully applicable to RAG chains; manual construction of `ModelSignature` is valid but more verbose and error-prone. Option d is wrong — signatures are not auto-inferred at serving time; they must be embedded in the logged model artifact, and passing `signature=` is actively recommended (not deprecated) in current MLflow docs.

**Q160: A Generative AI Engineer configures a Unity AI Gateway route to accept OpenAI-compatible chat completion requests and forward them to a Databricks Model Serving endpoint. Which `route_type` value should they set?**

O: a) `route_type: "serving/v1/predict"` — this is the generic predict route for all model types.
   b) `route_type: "chat_completion"` — this is the legacy short-form identifier for chat routes.
   c) `route_type: "llm/v1/chat"` — this configures the route to accept and validate OpenAI chat completion payloads (`{"messages": [...]}`).
   d) `route_type: "model/v2/chat"` — this is the Databricks-native route type for agent endpoints.

A: **c**. The Unity AI Gateway (formerly MLflow AI Gateway) defines three standard `route_type` values: `"llm/v1/chat"` for OpenAI-style chat completion requests (`{"messages": [...], "max_tokens": ...}`), `"llm/v1/completions"` for legacy text completion requests (`{"prompt": "...", "max_tokens": ...}`), and `"llm/v1/embeddings"` for embedding requests (`{"input": "..."}`). Setting `route_type: "llm/v1/chat"` tells the gateway to validate incoming payloads against the chat schema and route them to the configured backend endpoint. Option a is wrong because `"serving/v1/predict"` is not a valid `route_type` — it resembles a generic serving path but does not correspond to any AI Gateway route type. Option b is wrong because `"chat_completion"` is not a valid `route_type` value; the correct form includes the versioned namespace prefix `"llm/v1/"`. Option d is wrong because `"model/v2/chat"` is not a recognised AI Gateway route type — Databricks does not use a `model/v2/` namespace for gateway configuration.

**Q161: A Generative AI Engineer is sizing a provisioned throughput Foundation Model API endpoint. The application requires 10,000 tokens per second at peak. What is the correct unit used to size a provisioned throughput endpoint on Databricks?**

O: a) DBU-hours per day — the number of Databricks compute billing units consumed per day at the target load.
   b) Tokens per second (TPS) — the number of input plus output tokens the endpoint can process per second at the committed throughput level.
   c) Concurrent requests per minute — the maximum number of simultaneous in-flight requests the endpoint handles before queuing.
   d) GB of model weights per endpoint — the total serialised size of the model, which determines the underlying instance type and throughput capacity.

A: **b**. Databricks Foundation Model API provisioned throughput endpoints are sized in **tokens per second (TPS)** — the engineer specifies the peak combined input+output token throughput the endpoint must sustain. The platform provisions dedicated compute capacity to guarantee that TPS level; requests arriving at a rate above the provisioned TPS are throttled. This is distinct from pay-per-token (where you are billed per token consumed with no throughput guarantee) and from custom model serving (which is sized by workload size + concurrency). Option a is wrong because DBU-hours are a billing and cost metric, not a throughput sizing parameter — you do not configure a provisioned throughput endpoint by entering DBU values. Option c is wrong because "concurrent requests per minute" is a rate metric that describes request-level parallelism; provisioned throughput is sized in tokens (the unit that reflects actual model computation load), not request counts. Option d is wrong because model weight size affects which hardware Databricks selects internally, but it is not an input parameter exposed to the engineer when sizing a provisioned throughput endpoint — TPS is the directly configured value.

**Q162: Which TWO of the following steps must be completed before a Databricks Model Serving endpoint can successfully serve a model registered in Unity Catalog?**

   A. The model must be exported to ONNX format before Model Serving can load it.
   B. The model must be registered to Unity Catalog with a three-part name (`catalog.schema.model_name`) and have at least one version whose artifact is fully materialised and in a `READY` state.
   C. The Unity Catalog metastore must be detached and reattached from the workspace after every new model version registration.
   D. The serving endpoint configuration must explicitly reference the Unity Catalog model name and the specific version (or alias) to deploy — the endpoint is not created automatically when a model is registered.
   E. The model's source Delta table must be run through `OPTIMIZE` and `VACUUM` before the serving endpoint can access the model artifact.

A: **B and D**. A Model Serving endpoint cannot exist without a registered model to point to: the model must exist in Unity Catalog under a three-part name (`catalog.schema.model_name`) and have a version with a fully available artifact (`READY` state) — a model version that is still uploading or in an error state cannot be served. Separately, the serving endpoint must be explicitly created (via the Serving UI, REST API, MLflow Deployments SDK, or `databricks.agents.deploy()`) with configuration referencing the UC model name and version (or alias such as `"Champion"`); registering a model to UC does not automatically create or update any serving endpoint. Option A is wrong because Databricks Model Serving loads MLflow-flavoured models (PyFunc, LangChain, sklearn, etc.) directly — ONNX export is one serialisation option for PyTorch/TF models but is not a general requirement. Option C is wrong because metastore detach/reattach is a workspace-level administrative operation unrelated to model serving; it is never required as part of the model registration or deployment workflow. Option E is wrong because the serving endpoint accesses the model artifact from the MLflow artifact store (typically cloud object storage), not from a Delta table — `OPTIMIZE`/`VACUUM` are Delta table maintenance commands with no relevance to loading a serialised model file.
