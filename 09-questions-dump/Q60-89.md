**Q60: A Generative AI Engineer wants an LLM to always return its output as valid JSON with specific keys ("summary", "sentiment", "confidence") so downstream code can parse it reliably. Which prompting approach is MOST effective?**

O: a) Ask the question conversationally and hope the model infers the right format.
   b) Explicitly specify the exact JSON schema and keys in the prompt, and include an example of correctly formatted output.
   c) Increase the temperature to encourage varied formatting so the best one can be chosen later.
   d) Avoid giving any formatting instructions since LLMs infer format automatically.

A: **B**. Explicitly specifying the required schema/keys and providing a formatted example is the most reliable way to elicit consistently structured output that downstream code can parse reliably without manual error handling. Seeing a concrete example of the desired JSON format teaches the LLM the exact output structure and prevents format deviations that would break downstream code. **A is wrong because** asking conversationally and hoping the model infers the right format is unreliable—without explicit schema specification, the LLM may return prose, bullet lists, or valid but differently-keyed JSON that breaks parsing. **C is wrong because** increasing temperature encourages the model to deviate from instructions and produce more varied (and likely malformed) outputs—the opposite of consistent structured formatting. **D is wrong because** LLMs do not automatically infer that output should be formatted as JSON without explicit instruction; they default to natural language prose unless instructed otherwise in the prompt.

**Q64: A Generative AI Engineer is applying a chunking strategy for a document structure consisting of long legal contracts with numbered clauses. Which chunking approach is MOST appropriate given this structure?**

O: a) Chunk purely by fixed character count with no regard for clause boundaries, frequently splitting a clause mid-sentence.
   b) Chunk along natural clause/section boundaries so each chunk contains a complete, self-contained clause or logical unit.
   c) Put the entire contract into a single chunk regardless of length.
   d) Chunk by random sampling of paragraphs across different contracts.

A: **B**. Respecting the document's natural structure (clause/section boundaries) for chunking preserves semantic completeness and legal context, which is especially important for legal text where clauses are self-contained, legally binding units of meaning. Splitting mid-clause destroys the semantic coherence that courts and lawyers expect; a retrieved mid-clause chunk provides only partial information and risks misrepresenting legal obligations. **A is wrong because** fixed character-count chunking with no regard for clause boundaries frequently splits clauses mid-sentence, returning fragments that are incomplete and potentially legally misleading to the application's users. **C is wrong because** storing the entire multi-thousand-token contract as a single chunk overwhelms the LLM's context window, leaves no room for other context or the query, and makes it impossible to retrieve specific relevant clauses—the retrieval step becomes useless. **D is wrong because** random sampling of paragraphs across different contracts has no semantic meaning—random selections combine unrelated legal obligations, violate document structure, and would produce nonsensical results.

**Q66: A Generative AI Engineer needs to extract both tables and narrative text from a mix of Word documents, PDFs, and HTML files with minimal custom parsing code per format. Which Python library choice is BEST?**

**Q69: A Generative AI Engineer is considering adding a cross-encoder re-ranker to their RAG pipeline. Their current pipeline retrieves the top 5 chunks with ANN search, and p95 latency is already at the 4-second SLA limit. What is the MOST important consideration before adding re-ranking?**

O: a) Re-ranking always reduces latency since it filters chunks, so it will automatically improve SLA compliance
   b) Cross-encoder re-ranking adds meaningful per-query compute cost and latency (it scores every candidate pair), which could push the pipeline over the SLA limit — the engineer must benchmark the latency impact and consider whether a lighter bi-encoder or threshold-based approach is a better fit
   c) Re-ranking is only applicable when retrieving more than 20 chunks; at top-5 it has no effect
   d) Re-ranking eliminates the need for the ANN retrieval step, so latency would decrease overall

A: **B**. Cross-encoder re-rankers are computationally expensive — they run a full model forward pass for every (query, chunk) candidate pair. In a pipeline already at its latency ceiling, adding re-ranking risks SLA violations. The engineer should benchmark the added latency and evaluate lighter alternatives (bi-encoder re-ranking, simpler scoring heuristics) before committing. Option A is wrong — re-ranking adds latency, it does not reduce it. Option C mischaracterises when re-ranking applies. Option D is wrong — re-ranking refines results from retrieval; it does not replace retrieval.

**Q72: A Generative AI Engineer has a baseline LLM response that is too verbose and formal for a casual consumer chatbot. Which prompt adjustment would MOST directly move the output toward the desired casual, concise style?**

O: a) Add instructions specifying a casual tone and a maximum response length, optionally with a few-shot example of the desired style.
   b) Increase the max_tokens parameter significantly.
   c) Remove the system prompt entirely.
   d) Switch to a model with a longer context window.

A: **A**. Explicit tone and length instructions, reinforced with a few-shot example, directly guide the LLM toward the desired casual and concise style — a more reliable lever than parameters like max_tokens or context length. The system prompt and examples teach the model *how* to speak, not just *how much* to generate. When the baseline is verbose and formal, adjusting model parameters alone (max_tokens, temperature, context window) changes *quantity* or *confidence* but not inherent *style*—the model will remain verbose unless instructed otherwise. **B is wrong because** increasing max_tokens makes the problem worse—it gives the model *more* room to be verbose, not less. **C is wrong because** removing the system prompt entirely removes any guidance about desired tone; the model would default to its base training, which likely has no style preference and may remain formal/verbose. **D is wrong because** a longer context window does not affect output style; it only increases the amount of context available for grounding—style is controlled by prompt instructions, not architectural parameters.
